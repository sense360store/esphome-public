# CTF Security Audit — sense360store/esphome-public — 2026-07-03

Read-only capture-the-flag audit. The auditor plays attacker against the owner's
own public repository and attempts to capture each flag using only non-destructive,
read-only means. A captured flag is a real vulnerability. No file in this repository
was edited, staged, committed, or deleted during the audit phase except this report.
No release was created, no credential was rotated or printed in full, and no
hard-stop file (release/publish/signing workflow, credential scanner, signing key,
manifest) was modified.

Method summary: local `git log`/`grep`/`sed` over a clean checkout of `main`
(`93feb7c`), read-only inspection of every workflow and the credential-handling
scripts, and download plus `strings`/offset inspection of already-published GitHub
Release binaries. Published binaries were fetched read-only from the Releases CDN.

Baseline: no prior `CTF-AUDIT-*` file exists, so this is the first CTF baseline for
this repo. It is cross-referenced against the prior narrative audit
`docs/security/SECURITY-AUDIT-2026-06.md` (findings H1/H2). Future re-runs diff
their scoreboard against the table below.

## Findings

### FLAG-BIN-SECRET — CAPTURED — Critical

Currently-published release binaries embed the exact default credential byte-strings
the release scanner is built to reject. The scanner and the secret-stripping step
that prevent this landed on 2026-06-10 (`scripts/check_firmware_default_credentials.py`,
first commit `9514803`, `SEC-ESP-BUILD-GATES-001`). Every published release predates
that gate and was never rebuilt or rescanned, so the vulnerable artifacts remain live
and downloadable.

Device-control defaults in the latest stable release
`Sense360-Ceiling-POE-AirIQ-RoomIQ-v1.0.6-stable.bin` (published 2026-06-09; also
present in v1.0.4). Read-only evidence, `strings -t x`:

```
0x41b8 se…ck   fallback_ap_password "shipped release default"  (scanner denylist :76)
0x41c9 se…lt   ota_password         "shipped release default"  (scanner denylist :66)
0x41de se…in   web_password         "shipped release default"  (scanner denylist :73)
```

The three sit in one contiguous credential block in flash:
`...Sense360_Setup.sense360setup.<apssid>.<fallback_ap_password>.<ota_password>.<web_password>.<hostname>...`
The OTA password permits persistent network firmware overwrite of any device still on
this default (ESPHome OTA on tcp/3232); the web password (username `admin`) authenticates
the on-device web UI; the fallback AP password grants RF-range captive-portal access.
The all-`a` API-key placeholder (scanner :46) was **not** present, so native-API
control via the placeholder key does not apply to these two stable images.

Scope across the currently-published `v1.0.0-preview` prerelease (10 assets, all
scanned):

- Fan builds (`*-FanPWM/FanDAC/FanRelay-*`, plus `Ceiling-POE-FanPWM/FanDAC`) embed
  the CI test-lane set `Te…23` (`wifi_password`, :89), `Te…rk` (`wifi_ssid`, :88),
  `te…rd` (`ota_password`, :67), `te…rd` (`web_password`, :74), and the burned
  literal `Se…ck` (`fallback_ap_password`, audit-H2, :84). Example:
  `Sense360-Ceiling-POE-VentIQ-FanPWM-RoomIQ-v1.0.0-preview.bin` — `TestPassword123`
  at 0x4335, `Sense360Fallback` at 0x435c.
- Non-fan previews (`Ceiling-POE-AirIQ-RoomIQ`, `Ceiling-POE-RoomIQ`,
  `Ceiling-POE-RoomIQ-LED`, `Ceiling-POE-VentIQ-RoomIQ-LED`) embed the shipped
  device-control defaults `se…lt` (ota), `se…in` (web) and the burned `Se…ck`
  fallback literal.

Read-only reproduction the owner can run:

```
curl -sSL -o v106.bin https://github.com/sense360store/esphome-public/releases/download/v1.0.6/Sense360-Ceiling-POE-AirIQ-RoomIQ-v1.0.6-stable.bin
python3 scripts/check_firmware_default_credentials.py v106.bin   # expect exit 1
```

Timeline (all release publish dates precede the 2026-06-10 gate): v1.0.0 2026-05-12,
v1.0.0-led-preview 2026-05-15, v1.0.0-preview 2026-06-02..04, v1.0.2 2026-06-07,
v1.0.4 2026-06-07, v1.0.5 2026-06-08, v1.0.6 2026-06-09.

Why Critical: the flag rubric scores "secret extractable from a shipped artifact" as
Critical, and these are device-control credentials (OTA overwrite, web auth, provisioning
AP) extractable from the newest published stable binary. Tempering context: the values
are already public (they live in the scanner denylist), so no novel secret is disclosed
by this report; the concrete exposure is field devices left on these versions and defaults.

### FLAG-SECRET-TREE — CAPTURED (history) — High

Working tree: SECURE. Every device credential in `packages/base/` is a `!secret`
reference (`api_encrypted.yaml:3`, `ota.yaml:3`, `wifi.yaml:7,20`,
`logging.yaml:14-15`); the only literal values are placeholders in `*.example` files;
`secrets.yaml` is gitignored (`.gitignore:2`); no `-----BEGIN`, `AKIA…`, or `ghp_/gho_`
patterns in tracked files.

Git history: CAPTURED. Shared fallback-AP password literals were committed and remain
permanently reachable. `git log --all -S 'Sense360Fallback' -- packages/base/wifi.yaml`
returns `e6a15dd SEC-ESP-FALLBACK-AP-001: move fallback-AP password to !secret`
(2026-06-06); the literals `Se…ck` and `se…oe` are readable at `e6a15dd^`. These are
genuine weak shared credentials that shipped on pre-fix units. This is the H2 finding
in `SECURITY-AUDIT-2026-06.md:37-42`: tree-remediated and on the forward denylist, but
the history dimension is openly documented as unremediable. Scored High per rubric
(real secret in history); forward exposure is controlled, affected field units require
re-flash.

### FLAG-SCANNER-BYPASS — CAPTURED (latent) — Low

Three structural gaps in the credential gate; none is a live leak today, each is a way
a future defaulted binary could pass.

1. Uncovered credential field: MQTT. `mqtt_password` / `airiq_mqtt_password` is neither
   stripped by `apply_release_secret_posture.py` (`TRANSFORMS` covers only
   api/ota/web/fallback) nor on the scanner denylist. It flows into a live `mqtt:` block
   via a plain `${substitution}` at `packages/features/airiq_basic_profile.yaml:43-47`,
   included by the stable release row `Ceiling-POE-AirIQ-RoomIQ`
   (`products/bundles/ceiling-poe-airiq-roomiq.yaml:141`). The default is empty
   (`airiq_basic_profile.yaml:29`) so nothing is baked now, but because it is a
   substitution and not `!secret`, a future non-empty shared default would compile
   silently, survive the strip, and pass the scanner — three controls miss it at once.
2. Denylist is fixed-value. `check_firmware_default_credentials.py` matches known
   byte-strings only (`:57-89`); any brand-new default value for even a covered class
   passes. The real primary control is the posture-strip; the scanner is only a backstop
   for enumerated historical values.
3. Path coverage. The scanner runs only inside `firmware-build-release.yml:519,592`.
   Binaries published by any other route (the pre-gate previews above, or a manual
   upload) never touch it. FLAG-BIN-SECRET is the concrete instance of this gap.

### FLAG-DEP-VULN — CAPTURED — Low

No `.github/dependabot.yml` exists in this repo, so there is no automated dependency
update stream for either the pip toolchain (`requirements-dev.txt`) or GitHub Actions.
Actions are, in practice, all manually pinned to full SHAs (see FLAG-ACTION-UNPINNED),
which mitigates the supply-chain half, but the absence of automated advisories/updates
is the hardening gap the flag names. No npm lockfile exists (Python repo), so `npm audit`
is not applicable.

### FLAG-DEFAULT-CRED — SECURE

The H1 regression check passes for the current source and build path. Release builds
provision only the two intentionally-public setup values
(`firmware-build-release.yml:287-289`, `wifi_ssid: Sense360_Setup` /
`wifi_password: sense360setup`) and then run `apply_release_secret_posture.py`
(`firmware-build-release.yml:262-274`), whose `TRANSFORMS` (`:67-92`) strip the
`api.encryption`, `ota.password`, `web_server` auth, and `fallback_ap_password` blocks so
released firmware ships unprovisioned. Post-conditions fail closed if a base file's shape
changed. Committed base YAML keeps `!secret`, so a product referencing a stripped key
fails compile rather than baking a value. The denylist scanner is the second layer. No
literal shared control credential exists on the current release build path. (The shipped
*artifacts* remain vulnerable — that is FLAG-BIN-SECRET, not a source-path regression.)

### FLAG-CI-PWN — SECURE

Only `validate.yml` (`:19-29`) and `compile-only.yml` (`:38-58`) are fork-reachable,
both via `pull_request` (not `pull_request_target`), both `permissions: contents: read`,
and neither references `secrets.*`. A fork PR therefore gets a read-only token and no
secrets even though it checks out untrusted code. `compile-only.yml` further gates its
secret-provisioning full-compile job behind `if: github.event_name == 'workflow_dispatch'`
(`:117`). `firmware-build-release.yml` is `release: [published]` (write-gated); all other
workflows are `workflow_dispatch` or `schedule`. No `pull_request_target`, `issue_comment`,
`issues`, or `workflow_run` triggers exist.

### FLAG-CI-INJECT — SECURE

No `run:` step interpolates an attacker-controlled event field into a shell. Inputs and
event fields are bound to `env:` and referenced as quoted `$VAR`
(`firmware-build-release.yml:80-82`; `bump-version.yml:93-104,128-140`;
`create-release.yml:116-119,172-181`). The one genuinely attacker-influenceable field, the
release body, is read from `GITHUB_EVENT_PATH` in Python and written to a file via
`Path(...).write_text(...)` (`firmware-build-release.yml:706-729`), never shell-interpolated.
Remaining `${{ }}`-into-`run:` uses reference only `matrix.*` (from committed
`config/webflash-builds.json`), `github.sha`, and pipeline-derived version/channel values.

### FLAG-ACTION-UNPINNED — SECURE

Every third-party `uses:` is pinned to a full 40-char commit SHA with a version comment,
including the two publish-path third-party actions
`sigstore/cosign-installer@d58896d…#v3.9.2` (`firmware-build-release.yml:623`) and
`softprops/action-gh-release@3bb1273…#v2.6.2` (`:739`). `actions/*` are likewise
SHA-pinned; the local composite is referenced by path. No floating tag anywhere.

### FLAG-TOKEN-SCOPE — SECURE

Workflow defaults are `contents: read`. The only `id-token: write` in the repo is the
single cosign signing job (`firmware-build-release.yml:564-566`, paired with
`contents: write` for the release upload). The three `contents: write` grants
(`bump-version`, `create-release`, release job) each map to a real write action and are
reachable only via `workflow_dispatch`/`release`. No `packages: write`, no `write-all`.

### FLAG-DEV-LEAK — SECURE

`dev/secrets.yaml.example` and `dev/device-template.yaml` carry only `REPLACE_ME_*`
placeholders that do not decode to real values; real bench files (`dev/secrets.yaml`,
`dev/*.local.yaml`) are gitignored (`.gitignore:92`). `dev/dev-overlay.yaml` is committed
empty and `scripts/check_dev_harness_guard.py:41-62` fails CI on any functional line.
`check_dev_harness_guard.py:83-101` walks every string in every `config/*.json` and fails
on any `dev/`-selecting path; the release matrix is declaration-driven from
`config/webflash-builds.json` with no `dev/` glob. No build or release path can select
`dev/`.

## Regression note

No prior CTF baseline exists, so no SECURE→CAPTURED regression can be computed this run.
Relative to `SECURITY-AUDIT-2026-06.md`: the H1 source-path finding is confirmed
remediated (FLAG-DEFAULT-CRED SECURE) and H2 is confirmed as the documented, unremediable
history exposure (FLAG-SECRET-TREE). The new observation this run is that the H1/H2
remediation gates only *future* builds, while every *already-published* binary still
carries the defaults (FLAG-BIN-SECRET) — this was outside the June audit's source-tree
scope and is the material item for the owner.

## Gated remediation appendix (proposals only — nothing applied)

- FLAG-BIN-SECRET — **GATED** (touches releases/publish). Options for owner decision:
  (a) delete or replace the credential-bearing assets on the affected Releases and
  re-publish rebuilt, scanner-clean binaries; (b) at minimum, mark the pre-2026-06-10
  releases as affected in their release notes and instruct field devices to re-flash the
  latest gated build and rotate device credentials. Any asset rebuild/replace runs through
  the signing/publish path and must not be automated without owner sign-off. A safe,
  non-mutating first step is to run `check_firmware_default_credentials.py` over every
  published asset to produce the authoritative affected-list.
- FLAG-SECRET-TREE (history) — **GATED / accepted risk.** History rewrite of a public repo
  is destructive and generally infeasible; the burned literals are already on the forward
  denylist. The actionable item is device-side: ensure any field unit that ran a pre-fix
  build has its fallback-AP/OTA/web credentials rotated by re-flash.
- FLAG-SCANNER-BYPASS — **SAFE TO AUTOMATE** (does not touch the hard-stop set): convert
  `airiq_mqtt_password` to `!secret` (or add it to the posture-strip `TRANSFORMS` and the
  scanner denylist) so the MQTT field is covered by the same three controls as the other
  credential classes. Consider extending the scanner from a fixed denylist toward an
  entropy/format check for the covered classes. These edit `scripts/`/`packages/` only,
  not release/publish/signing logic — but still require an explicit remediation-phase
  approval per the loop rules.
- FLAG-DEP-VULN — **SAFE TO AUTOMATE**: add `.github/dependabot.yml` covering
  `github-actions` and `pip` ecosystems.
