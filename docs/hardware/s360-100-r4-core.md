# S360-100-R4 — Sense360 Core Hardware Reference

## Purpose

This document is the schematic-backed pin, connector, and net reference for the
**Sense360 Core** main board, revision **R4** (`S360-100-R4`). It exists so that
firmware audits — in particular the upcoming Release-One YAML audit — have a
single, citable source of truth for what the silicon and copper actually do.

This document is hardware reference only. It does **not** by itself prove that
any ESPHome YAML in this repository is correct. Firmware mappings must be
verified separately by comparing this document against the package and product
YAML (see [Firmware mapping notes](#firmware-mapping-notes)).

## Main components

- **U6** — `ESP32-S3-WROOM-1-N16R8` MCU module (16 MB flash, 8 MB octal-SPI PSRAM).
- **U3** — `SX1509BIULTRT` 16-channel I²C GPIO / PWM expander.
- **U1** — `RT8059GJ5` step-down buck converter (5 V → 3.3 V).
- **U2A** — `74LVC1G07SE-7` open-drain buffer used as `LED_DATA` driver / level shifter.
- **J8** — USB-C 2.0 receptacle (data + VBUS).
- **J2** — `PoE_ACDC` 2-pin power inlet (fed by external PoE PSU module S360-410).
- **Q2** — `Si2319CDS` p-channel MOSFET, ideal-diode-OR between USB VBUS and PoE_ACDC into +5 V.
- **Q4** — `SI2302S` n-channel MOSFET on the LED level-shifter section.
- **R8** — 330 Ω series resistor on the WS2812B data line.
- **R12 / R13 / R21 / R22** — 10 kΩ pull-ups (reset, expander INT, I²C SDA/SCL).
- **R5 / R6** — 70.6 kΩ / 15.1 kΩ buck feedback divider (per schematic note,
  computed Vout ≈ 3.4 V due to manufacturer compensation, in practice ≈ 3.3 V).
- **L1** — 2.2 µH buck inductor.
- **D5** — yellow LED, +5 V rail indicator.
- **D6** — blue LED, +3.3 V rail indicator.
- **D12** — `1N5819WS` Schottky diode (USB VBUS path).
- **SW3** — Boot/IO0 tactile switch (pulled to +3.3V via R40 10 k).
- **SW4** — Reset tactile switch (drives `EN` via R12 10 k pull-up + C44 1 µ).
- **TP10 / TP13 / TP14** — JTAG test points (`MTMS`, `MTDO`, `MTDI`).
- Module connectors: `J3` (WS2812B LED), `J4` (Relay), `J6` (12 V PWM fan),
  `J7` (GP8403 fan), `J9` (AirIQ module), `J10` (Presence/Comfort module —
  RoomIQ), `J13` (FAN), `J15` (TRIAC fan module).

## I2C bus

| Net       | ESP32 pin | Pull-up    | Devices / connectors that share the bus                                              |
| --------- | --------- | ---------- | ------------------------------------------------------------------------------------ |
| `I2C_SDA` | IO48      | R22 10 k   | SX1509 (U3, on-board), J7 GP8403 fan, J9 AirIQ, J10 RoomIQ                           |
| `I2C_SCL` | IO45      | R21 10 k   | SX1509 (U3, on-board), J7 GP8403 fan, J9 AirIQ, J10 RoomIQ                           |

The SX1509 GPIO/PWM expander sits on the same shared I²C bus as the external
modules. Its `INT` line is `expander_int` (ESP32 IO17) with R13 10 k pull-up,
and its `RESET` is driven by `RST1`.

## UART / radar-related signals

The Core board exposes three logical UARTs at the connector level:

| Logical UART    | ESP32 pins                  | Destination                                          | Purpose                                  |
| --------------- | --------------------------- | ---------------------------------------------------- | ---------------------------------------- |
| UART0 (boot log)| TXD0 (pin 37) / RXD0 (pin 36) | J7 GP8403 fan (also used as boot log on USB)       | GP8403 DAC fan driver UART               |
| `Hi-Link`       | IO2 (TX) / IO1 (RX)         | J10 RoomIQ Pin 7 / Pin 6                             | Hi-Link mmWave radar UART (RoomIQ)       |
| `SEN0609`       | IO5 (TX) / IO4 (RX)         | J10 RoomIQ Pin 4 / Pin 3                             | DFRobot SEN0609 radar UART (RoomIQ)      |

`out(gpio6)` (ESP32 IO6 → J10 pin 5) is an additional auxiliary line carried
to the RoomIQ connector but is not, on this sheet, dedicated to a UART.

## Open questions / verification needed

These items are explicitly out of scope for HW-002 to resolve. They are the
inputs to the later Release-One YAML audit PR.

1. **Which ESP32 GPIO drives `TRI_GPIO1` and `TRI_GPIO2` on J15?** The
   schematic shows these nets at the connector and at the SX1509 (U3) side, but
   does not show a direct route to a free ESP32 pin. Most likely both signals
   are driven from SX1509 expander I/Os, not from the ESP32 directly.
2. **Release-One YAML uses `GPIO5` / `GPIO6` for FanTRIAC.** The schematic
   shows `IO5 = SEN0609_TX` and `IO6 = out(gpio6)`, both wired to the J10
   RoomIQ connector. These cannot also be the TRIAC gate / zero-crossing pins.
   The audit must determine the correct physical mapping before any YAML
   change.
3. ~~**LED ring YAML uses `GPIO14`.**~~ **Resolved by HW-010.** The
   ceiling LED package
   ([`packages/hardware/led_ring_ceiling.yaml`](../../packages/hardware/led_ring_ceiling.yaml))
   now binds `led_data_pin: GPIO38`, matching `LED_DATA = IO38` on this
   schematic (`IO14` remains `SCS`, peripheral SPI chip-select).
   HW-010 did not add `LED` to the Release-One config string and did
   not promote Sense360 LED to WebFlash-shippable. The wall LED package
   ([`packages/hardware/led_ring_wall.yaml`](../../packages/hardware/led_ring_wall.yaml))
   and the legacy S3 Core package
   ([`packages/hardware/sense360_core_ceiling_s3.yaml`](../../packages/hardware/sense360_core_ceiling_s3.yaml))
   were intentionally not changed — neither has Core-side schematic
   evidence proving the same `LED_DATA` path.
4. **`AirQ_Led` / `AirQ_Status_Led` net naming.** Both nets are still labelled
   with the legacy `AirQ` prefix. When a VentIQ (S360-211) module is plugged
   into J9 instead of an AirIQ (S360-210), confirm whether the same physical
   lines are reused and whether the firmware should expose them under a
   different friendly name.
5. **Is `Sense360 LED` (S360-300) optional or part of the default Release-One
   hardware?** The presence of J3 on the Core board does not by itself answer
   whether a Release-One Ceiling unit ships with the WS2812B ring populated.
6. **Which connector is physically used by the S360-410 PoE PSU module?**
   J2 carries the `PoE_ACDC` net, but the PoE magnetics and RJ45 live on the
   PSU module. Confirm the cable / harness between the PoE PSU and J2.
7. **Confirm `IO10` net label.** The label next to ESP32 module pin 18 (IO10)
   is not unambiguously legible in the PDF.
8. **Confirm exact pin-to-test-point assignment among `IO39 / IO40 / IO41 /
   IO42`** for `MTMS` / `MTDO` / `MTDI` / Peripheral. The present mapping in
   this document is based on the visible labels but is marked **verify**.
9. **Confirm pin-by-pin numbering for J6 (13 pins) and J10 (12 pins).** The
   net list is correct; the 1-to-N pin order on each header should be checked
   against the silkscreen.
10. **The pin/net example table embedded in the HW-002 task brief differs
    from the schematic** for several signals (notably MISO/MOSI/SCLK/SCS,
    FAN, Relay, expander_int, AirQ_Led, AirQ_Status_Led, LED_DATA, I2C_SDA,
    I2C_SCL, ALS_INT, RST). This document deliberately follows the schematic
    rather than the brief; the divergence is captured here so that the audit
    PR can decide whether the brief, the YAML, or both need follow-up.
