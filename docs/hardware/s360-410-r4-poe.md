# S360-410-R4 PoE PSU Pin / Package Mapping Audit (HW-PINMAP-410)

## Status

**Status: partial — schematic evidence available; package
reconciliation, PoE PD controller / magnetics / buck / isolated
DC/DC / harness identity evidence pending.**

This document is the **HW-PINMAP-410** audit record for the
`Sense360 PoE PSU` board, revision **R4** (`S360-410-R4`). It exists
to close the documentation gap recorded at
[`docs/hardware/hardware-artifact-policy.md` Follow-up PR sequence #8](hardware-artifact-policy.md#follow-up-pr-sequence)
and at
[`docs/hardware/board-readiness-matrix.md` Follow-up PR sequence #6](board-readiness-matrix.md#follow-up-pr-sequence),
which together call for a per-board pin/package mapping audit
to close the schematic-pending caveat preserved in
[`docs/release-one-hardware-audit.md` Findings → PoE PSU](../release-one-hardware-audit.md#findings).

When HW-PINMAP-410 originally landed, the audit was
`pending — schematic/design evidence required`: no module-side
`S360-410-R4` schematic existed in this repository, and no curated
artifact index built from it had been committed. **HW-ASSETS-410
has since landed as PR #516** — the module-side schematic PDF is
committed at
[`docs/hardware/schematics/S360-410-R4.pdf`](schematics/S360-410-R4.pdf)
(975,137 bytes; SHA256
`4a8b7a3b2a89006a9332eaa486743f687aaedc4b6bb807c6b25670f742ac2414`;
byte-identical to the upload) and the curated artifact index lives
at
[`docs/hardware/artifacts/S360-410-R4.md`](artifacts/S360-410-R4.md).
**HW-PINMAP-410-FOLLOWUP** (this revision) consumes that evidence
to record a schematic-backed reconciliation view. It does **not**
resolve the part-identity / silkscreen / BOM / isolation /
compliance / harness disagreements it surfaces — those are owed
to separate, evidence-bearing follow-up PRs (BOM cross-check;
silkscreen photographs or KiCad PCB source; bench / load /
PoE-link-up / thermal / EMI / isolation evidence;
`PACKAGE-POE-410-001`; HW-002 OQ#6 / `S360-100-BENCH-001`
J2-harness closure).

The schematic-backed evidence on file proves the **logical**
PoE-input topology (`LAN_CON1 RJP-003TC1(LPJ4112CNL)` integrated
10/100 BASE-TX magnetics / RJ45 module with Bob-Smith bridge,
`2x1000pF/2KV` mid-span caps, `2x75 Ohm` terminations, and
`C3 1nF` shield-to-`Lan_earth` bridge), the **PoE PD controller
topology** (`U1 = TPS2378DDAR(HSOIC-8)` with `R1 24.9k` detection
signature, `R2 1.27k` classification programming labelled
`Class=0 (0.44 to 12.95W)`, `D1 SMAJ58A` TVS, `C2 15uF` CBULK,
`R5 0.03R` RTN current sense), the **buck stage topology** (`U2 =
TX4138(ESOIC-8)` with `R3/R4 9.1k` paired ILIM network, `L1 33uH`
inductor, `D2 ss510` Schottky catch, `C6 470u` output bulk, and a
`R7 10.5k` (Rd) / `R8 56.2k` (Rc) feedback divider with the
on-sheet formula `Vout = 0.8 * (1 + Rc/Rd)`), the **isolated
DC/DC topology** (`DCDC1 = F0505S-2WR2(SIP-7)` 5 V → 5 V isolated
module with `AM1D-0505S-NZ` annotated as alternate; pinout
`Vin+` / `Vin-` / `Vout-` / `Vout+`), the **output topology**
(`J3` 2-pin "Connection to Cores" with pin 1 = `+5VP`, pin 2 =
`GND`, plus `C8 22u` bulk and `R9 1k` bleed), the **buck-output
status indicator** (`D3` green LED on the buck output), the
**mounting-hole capture** (`H1`–`H4` each labelled `Earth` at the
symbol layer), and the **net topology across the isolation
boundary** (pre-isolation: `Spare1` / `Spare2` / `Lan_earth` /
`PoE_Power` / `PoE_SW` / `Sw_Vin_Poe` / `Earth`; post-isolation:
`+5VP` / `GND`, re-referenced — the post-isolation `GND` is **not**
the same node as the pre-isolation `Earth` / `Lan_earth` rail).

The schematic does **not** prove silkscreen pin-1 orientation on
`J3`, does **not** prove the physical connector identity /
current rating of `J3`, does **not** prove the per-component
ratings implied by the value-field strings (`R1 24.9k` /
`R2 1.27k` / `D1 SMAJ58A` / `C2 15uF` / `R5 0.03R` tolerance /
power / package; `R3/R4 9.1k` / `L1 33uH` / `D2 ss510` / `C6 470u`
/ `R7 10.5k` / `R8 56.2k` ratings; `F0505S-2WR2` isolation /
Hi-pot / leakage), does **not** prove `J3` connector
keying / polarity, does **not** prove `LAN_CON1` magnetics-module
mating, does **not** prove PCB-level electrical bonding of
`H1`–`H4` to `Lan_earth` / RJ45 shield, does **not** prove PoE
link-up against an 802.3af or 802.3at PSE, does **not** prove
signature / classification behaviour in hardware, does **not**
prove load regulation / cold-start inrush / thermal rise / EMI /
EMC behaviour, does **not** prove insulation resistance / Hi-pot /
earth-continuity / leakage for the assembled board, and does
**not** resolve the **part-identity disagreement** between the
package header in
[`packages/hardware/power_poe.yaml`](../../packages/hardware/power_poe.yaml)
(line 6 `Ag9712M, Silvertel Ag9700, or similar` — whole-module
PoE-PSU hint) and the schematic-shown discrete topology
(`TPS2378DDAR(HSOIC-8)` PoE PD controller + `TX4138(ESOIC-8)`
buck + `F0505S-2WR2(SIP-7)` isolated DC/DC with
`AM1D-0505S-NZ` annotated alternate + `RJP-003TC1(LPJ4112CNL)`
magnetics module). That resolution is owed to BOM evidence under
`PACKAGE-POE-410-001`. The HW-002 Open Question #6 J2-harness
identity closure is owed independently to bench / manufacturing
evidence tracked on
[S360-100-BENCH-001](s360-100-r4-core.md#s360-100-bench-001-status).
**Release-One and the LED preview continue to consume `S360-410`

Part field reads `RJP-003TC1(LPJ4112CNL)`. Schematic legend:
"Single 10/100 BASE-TX Filter Connector Module".

|---:|---|
| 0 | `Shield` |
| 1 | `TD+` |
| 2 | `TD-` |
| 3 | `RD+` |
| 4 | `TCT` |
| 5 | `RCT` |
| 6 | `RD-` |
| 7 | `GND/NC` |

Adjacent labels on the schematic name the magnetics-side pin

## Compliance / safety status

The committed schematic is **schematic evidence**, not
**compliance evidence**. This audit does **not**:

- claim **IEEE 802.3af / 802.3at compliance.** The
  schematic-visible `Class=0 (0.44 to 12.95W)` annotation
  declares the intended PD classification on the design; it is
  **not** a compliance test against a PSE.
- claim any **isolation rating.** The schematic shows an
  `F0505S-2WR2(SIP-7)` (with `AM1D-0505S-NZ` annotated alternate)
  isolated DC/DC module providing the galvanic-isolation barrier
  between the post-buck `Sw_Vin_Poe` and the Core-facing `+5VP` /
  `GND` output. The isolation rating is a BOM / datasheet claim;
  as-built insulation resistance / Hi-pot / leakage is bench
  evidence.
- claim any **earth-continuity.** The schematic shows four
  mounting holes `H1`–`H4` each labelled `Earth`, but PCB-level
  bonding to the RJ45 shield through plated holes / pads is
  **not** derivable from the schematic alone.
- claim any **thermal** behaviour. Continuous-load thermal rise
  of `U1` (TPS2378), `U2` (TX4138), and `DCDC1` (F0505S-2WR2) is
  **not** schematic-derivable.
- claim any **EMI / EMC** behaviour. Conducted / radiated
  emissions and immunity are **not** schematic-derivable.
- claim any **PoE link-up / bench load** behaviour. Observed
  link-up against an 802.3af or 802.3at PSE, observed signature
  / classification, observed load regulation across the rated 5
  V output range, and observed cold-start inrush are bench
  evidence and are not on file.

PoE is SELV. **`S360-410` is not in scope for COMPLIANCE-001**
(the mains-voltage UK / EU compliance tracker scopes `S360-320`
FanTRIAC and `S360-400` 240 V PSU only). A separate IEEE 802.3af
/ 802.3at PoE-compliance evidence class would be required before
any new PoE-410-explicit product surface, but that surface is
itself gated by `HW-PINMAP-410-FOLLOWUP` + `PACKAGE-POE-410-001`
+ HW-002 OQ#6 / S360-100-BENCH-001 + a `S360-410`
`schematic_status: verified` JSON-only PR + a Release-One
caveat-closure PR. **None of those happens in this PR.**
