# S360-312-R4 DAC / Analog Fan Driver Pin / Package Mapping Audit (HW-PINMAP-312)

## Status

**Status: partial — schematic evidence available; package reconciliation pending.**

This document is the **HW-PINMAP-312** audit record for the
`Sense360 DAC` board, revision **R4** (`S360-312-R4`). It exists to
close the documentation gap recorded at
[`docs/hardware/hardware-artifact-policy.md` Follow-up PR sequence](hardware-artifact-policy.md#follow-up-pr-sequence)
and at
[`docs/hardware/board-readiness-matrix.md` Follow-up PR sequence](board-readiness-matrix.md#follow-up-pr-sequence),
which together call for a per-board pin / package mapping audit
**before** any `FanDAC`-bearing package, product, build, or
WebFlash work can proceed.

HW-PINMAP-312 has a committed module-side schematic to work from: the
curated artifact index
[`docs/hardware/artifacts/S360-312-R4.md`](artifacts/S360-312-R4.md)
landed under HW-ASSETS-003 with the schematic PDF at
[`docs/hardware/schematics/S360-312-R4.pdf`](schematics/S360-312-R4.pdf).
This audit consumes that evidence to record a schematic-backed
reconciliation view. It does **not** resolve the firmware-side
disagreements it surfaces — those are owed to separate,
evidence-bearing follow-up PRs.

### Voltage-rail discrepancy

The single most important pin-map discrepancy carried by this
audit. The Core-side `J7` capture in
[`s360-100-r4-core.md` §J7](s360-100-r4-core.md#j7--gp8403-fan-connector-6-pin)
records pin 1 as `+5V`. The module-side `J1` capture labels pin 1
as `+3.3V`, and the on-module MT3608 boost converter's `IN` pin
is also labelled `+3.3V`. Three schematic-internally-consistent
possibilities exist:

1. **The Core-side `J7` capture is incorrect** and the supply
   delivered to the connector is actually `+3.3V`. In this case,
   [`s360-100-r4-core.md` §J7](s360-100-r4-core.md#j7--gp8403-fan-connector-6-pin)
   must be updated and the Core `J7` pin-1 row corrected against
   silkscreen.
2. **The module-side schematic mislabels the rail** and the board
   actually accepts `+5V` (consistent with boosting to `+12V` via
   MT3608 at a more comfortable voltage ratio). In this case, the
   module schematic and the artifact-index re-restatement are both
   wrong on the pin-1 label, and only the rail label needs
   correcting (the on-board behaviour would already be a `+5V`-in
   board).
3. **The Core and module agree on connector position 1 only by
   accident on the harness**, and the actual rail delivered to the
   module differs from what either schematic claims. In this case
   the harness conductor mapping needs separate verification.

**This is a wiring-relevant flag and must be resolved against
physical silkscreen evidence before any FanDAC-bearing config
ships.** This audit records the discrepancy. It does **not** edit
[`s360-100-r4-core.md`](s360-100-r4-core.md) and does **not**
choose between the three possibilities.

### UART pins on `J1` pins 4–5

The module-side `J1` labels pins 4 / 5 as `UART_RX` / `UART_TX`
and routes them on-board to the 4-pin Nextion display connector
`J7` (module side). The Core-side capture in
[`s360-100-r4-core.md` §Pin assignments](s360-100-r4-core.md#pin-assignments)
ties these nets to ESP32 `TXD0` (pin 37) and `RXD0` (pin 36) —
the same physical UART that
[`s360-100-r4-core.md` §UART buses](s360-100-r4-core.md#uart-buses)
records as **UART0**, **"also used as boot log on USB"**.

Two consequences belong to HW-PINMAP-312-FOLLOWUP:

1. **UART0 boot-log vs Nextion-display contention.** A future
   FanDAC-bearing config that consumes the Nextion display via
   this UART pair will share the wire with the ESP32 boot-log
   path. The behaviour at boot, during deep-sleep wakes, during
   firmware updates over UART, and during USB-attached debugging
   needs explicit bench evidence before the package YAML can bind
   a Nextion `display:` component or `uart:` peripheral on this
   pair.
2. **The package YAML does not currently bind these pins.**
   [`packages/expansions/fan_gp8403.yaml`](../../packages/expansions/fan_gp8403.yaml)
   declares only the two GP8403 DAC outputs and two `fan` speed
   entities; no UART peripheral, no Nextion display, no
   `uart:` block on the schematic-visible `UART_RX` / `UART_TX`
   pair. Whether to add a Nextion path is a separate package-side
   decision that the schematic alone cannot make.

This audit records the discrepancy. It does **not** edit
[`s360-100-r4-core.md`](s360-100-r4-core.md), does **not** edit
[`packages/expansions/fan_gp8403.yaml`](../../packages/expansions/fan_gp8403.yaml),
and does **not** choose whether a future FanDAC config consumes
the Nextion path.

### I²C address-selection scheme

The two DACs each have three address pins (`A0` / `A1` / `A2` on
`IC1`; `2A0` / `2A1` / `2A2` on `IC2`) routed through 4.7 kΩ
pull-ups (`R3` / `R5` / `R7` for `IC1`; `R4` / `R6` / `R8` for
`IC2`) to a 6-position DIP switch (`SW1` / `SW2`) tied to `GND`
(per
[`docs/hardware/artifacts/S360-312-R4.md` Reconciliation flags #3](artifacts/S360-312-R4.md#reconciliation-flags-raised-or-strengthened-by-this-schematic)).
Each DAC's I²C address is therefore **operator-selectable** at
assembly or in the field via the DIP switch.

[`packages/expansions/fan_gp8403.yaml`](../../packages/expansions/fan_gp8403.yaml)

### Header-comment claims vs schematic evidence

[`packages/expansions/fan_gp8403.yaml`](../../packages/expansions/fan_gp8403.yaml)
header comment lines 13–18 read:

```yaml
# Expansion Bus Connection:
#   Pin 4 (SDA) → GPIO39
#   Pin 5 (SCL) → GPIO40
#   Pin 2 (3.3V) → Power
#   Pin 1 (GND) → Ground
```

| Claim in header comment | Module `J1` capture (schematic) | Core `J7` capture (schematic) | Status |
|---|---|---|---|
| Pin 1 = `GND` | Pin 1 = `+3.3V` (per Module `J1` table) | Pin 1 = `+5V` (per Core `J7` table) | **Disagrees on both sides.** Module / Core agree pin 1 is a power rail, not ground (which is pin 6). |
| Pin 2 = `+3.3V` | Pin 2 = `I2C_SDA` | Pin 2 = `I2C_SDA` | **Disagrees on both sides.** The module / Core agree pin 2 is the I²C data line, not a power rail. |
| Pin 4 = `SDA` | Pin 4 = `UART_RX` | Pin 4 = `UART_RX` | **Disagrees on both sides.** The module / Core agree pin 4 is the UART receive line, not I²C SDA. |
| Pin 5 = `SCL` | Pin 5 = `UART_TX` | Pin 5 = `UART_TX` | **Disagrees on both sides.** The module / Core agree pin 5 is the UART transmit line, not I²C SCL. |
| `SDA → GPIO39` | I²C SDA is at Module `J1` pin 2, Core ESP32 `IO48` (per [`s360-100-r4-core.md` §I2C bus](s360-100-r4-core.md#i2c-bus)) | Same | **Disagrees with Core schematic.** `GPIO39` is not the Core I²C SDA pin. |
| `SCL → GPIO40` | I²C SCL is at Module `J1` pin 3, Core ESP32 `IO45` (per [`s360-100-r4-core.md` §I2C bus](s360-100-r4-core.md#i2c-bus)) | Same | **Disagrees with Core schematic.** `GPIO40` is not the Core I²C SCL pin. |

The header comment evidently describes a **different connector
shape** (a smaller, possibly 4-pin, I²C-only bus connector) and
**different ESP32 GPIOs** from the ones the schematic carries.
The active YAML body (the `i2c_id: ${fan_dac_i2c_id}` /
`address: ${fan_dac_address}` substitutions) does **not** depend
on the header comment's claims — `${fan_dac_i2c_id}` defaults to
the abstract bus name `i2c0`, which is supplied by whichever
parent Core package is included, and that abstract bus resolves
to ESP32 `IO48` / `IO45` per the Core schematic. The header
comment is therefore a **stale documentation block** rather than
an active firmware bug, but it is still a documentation-side
disagreement with the committed schematic evidence and is
recorded here.

This audit does **not** edit the header comment. The comment
update belongs to `PACKAGE-GAP-001` FanDAC slice.
