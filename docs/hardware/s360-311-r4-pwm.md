This document is the **HW-PINMAP-311** audit record for the
`Sense360 PWM` board, revision **R4** (`S360-311-R4`). It exists to
close the documentation gap recorded at
[`docs/hardware/hardware-artifact-policy.md` Follow-up PR sequence](hardware-artifact-policy.md#follow-up-pr-sequence)
and at
[`docs/hardware/board-readiness-matrix.md` Follow-up PR sequence #2](board-readiness-matrix.md#follow-up-pr-sequence),
which together call for a per-board pin/package mapping audit
**before** any `FanPWM`-bearing package, product, build, or
WebFlash work can proceed.

Unlike HW-PINMAP-310 (which is `pending — schematic/design evidence
required`), HW-PINMAP-311 has a committed module-side schematic to
work from: the curated artifact index
[`docs/hardware/artifacts/S360-311-R4.md`](artifacts/S360-311-R4.md)
landed under HW-ASSETS-003 with the schematic PDF at
[`docs/hardware/schematics/S360-311-R4.pdf`](schematics/S360-311-R4.pdf).
This audit consumes that evidence to record a schematic-backed
reconciliation view. It does **not** resolve the firmware-side
disagreements it surfaces — those are owed to separate,
evidence-bearing follow-up PRs.

## Schematic summary

This section restates only what is directly visible in the
committed schematic PDF
[`docs/hardware/schematics/S360-311-R4.pdf`](schematics/S360-311-R4.pdf)
and inventoried in
[`docs/hardware/artifacts/S360-311-R4.md` What the schematic appears to contain](artifacts/S360-311-R4.md#what-the-schematic-appears-to-contain).
It does **not** invent electrical characteristics, transistor
topology, timing guarantees, MOSFET specs, or pull-up assumptions
beyond what is labelled on the sheet.

- **Voltage Boost** section. `MT3608` boost converter (`U1`) with
  input inductor `L1 22 µH`, Schottky `D1 SS34`, feedback divider
  `R3 38 kΩ` / `R5 2 kΩ`, input cap `C1 22 µF / 10 V`, output cap
  `C2 22 µF / 10 V`, output indicator `R4 330 Ω` + blue LED `D2`.
  Boosts the Core-supplied `+5V` rail up to `+12V` for the fan
  outputs.
- **Fan outputs section** (labelled `"NINE 4pin FANs"` in the sheet
  title; see [§Open documentation questions](#open-documentation-questions)).
  **Four** 4-pin fan output connectors are present: `J1`, `J2`,
  `J4`, `J5`. Each fan connector has pins `+12V`, `GND`,
  `Pul_Cou*`, `TachPMW*` and is paired with one low-side N-channel
  MOSFET (`Q_NMOS_GSD` — `Q1..Q4`) on the `TachPMW*` gate path,
  with a 1 kΩ gate resistor (`R1`, `R2`, `R6`, `R7`); drain wired
  to the shared `TachIO` net; source to `GND`.
- **"From Core"** module-side connector `J3` — 13-pin. Pin / net
  table as labelled in the PDF:

  | Module-side `J3` pin | Net |
  |---:|---|
  | 1 | `+5V` |
  | 2 | `TachIO` |
  | 3 | `TachPMW1` |
  | 4 | `Pul_Cou1` |
  | 5 | `TachPMW2` |
  | 6 | `Pul_Cou2` |
  | 7 | `TachPMW3` |
  | 8 | `Pul_Cou3` |
  | 9 | `TachPMW4` |
  | 10 | `Pul_Cou4` |
  | 11 | `UART_RX` |
  | 12 | `UART_TX` |
  | 13 | `GND` |

- **"NEXTION DISPLAY"** connector `J6` (module-side) — 4-pin:
  `+5V`, `ESP32_TXD` / `UART_TX` (to display RX), `ESP32_RXD` /
  `UART_RX` (to display TX), `GND`. On-board route from module
  `J3` pins 11 / 12 to module `J6` UART pair.
- **Mounting holes** `H1`, `H2`, `H3`, `H4`.

## Connector / pin-map findings

### Module-side `J3` ↔ Core-side `J6` (13-pin harness)

The two connectors mate via a 13-pin Core ↔ module harness. The
module-side capture (above) is **schematic-confirmed** against the
committed S360-311-R4 PDF. The Core-side capture in
[`s360-100-r4-core.md` §J6](s360-100-r4-core.md#j6--12-v-pwm-fan-connector-13-pin)
records the net list but explicitly marks the 1-to-13 pin order as
**verify** against the silkscreen (
[`s360-100-r4-core.md` Open Questions #9](s360-100-r4-core.md#open-questions--verification-needed)).

| Element | Core side (`J6`) | Module side (`J3`) | Reconciliation status |
|---|---|---|---|
| Connector identity | `J6` — 12 V PWM fan connector (13-pin) — [§J6](s360-100-r4-core.md#j6--12-v-pwm-fan-connector-13-pin) | `J3` — "From Core" (13-pin) — schematic-confirmed | Naming mismatch is expected (each board labels its own connectors); 13-pin pin count matches. |
| Net list (signal nets only) | `+5V`, `GND`, `TachIO`, `TachPMW1..4`, `Pul_Cou1..4` | `+5V`, `GND`, `TachIO`, `TachPMW1..4`, `Pul_Cou1..4`, **`UART_RX`**, **`UART_TX`** | **Module side carries two additional nets (`UART_RX` / `UART_TX`) that the Core-side capture does not currently document.** See [§UART pins on `J3` pins 11–12](#uart-pins-on-j3-pins-11-12). |
| 1-to-13 pin order | **`verify`** against silkscreen — [`s360-100-r4-core.md` Open Questions #9](s360-100-r4-core.md#open-questions--verification-needed) | Schematic-only — order as labelled in the PDF (`+5V` → `TachIO` → `TachPMW1` → `Pul_Cou1` → … → `Pul_Cou4` → `UART_RX` → `UART_TX` → `GND`) | Schematic-only on the module side; **requires silkscreen verification** to confirm the harness wires `+5V` (Core J6) to `+5V` (Module J3 pin 1), and so on through to `GND` (Module J3 pin 13). Resolution belongs to the HW-PINMAP-311 follow-up + bench evidence. |

### UART pins on `J3` pins 11–12

The module-side `J3` labels pins 11 / 12 as `UART_RX` / `UART_TX`
and routes them on-board to the 4-pin Nextion display connector
`J6` (module side). The Core-side `J6` capture in
[`s360-100-r4-core.md` §J6](s360-100-r4-core.md#j6--12-v-pwm-fan-connector-13-pin)
does **not** currently list any UART pins on the 13-pin connector
— only `+5V`, `GND`, `TachIO`, and the eight `TachPMW*` /
`Pul_Cou*` signals (11 nets total) are recorded. Two
schematic-internally-consistent possibilities remain, **both in
scope for the HW-PINMAP-311 follow-up**:

1. **The Core-side `J6` capture is incomplete.** Pins 11 / 12 on
   the Core-side `J6` connector physically carry the UART pair
   (e.g. from `TXD0` / `RXD0` or a different ESP32 UART), and the
   capture in
   [`s360-100-r4-core.md` §J6](s360-100-r4-core.md#j6--12-v-pwm-fan-connector-13-pin)
   must be extended to record those two nets.
2. **The Nextion UART pair is delivered to the module via a
   separate harness, not via the 13-pin Core `J6` cable.** Pins
   11 / 12 on module-side `J3` would then be expected to be left
   unconnected (or to host different signals) on the actual
   13-pin cable. In this case the module-side schematic labels
   are misleading and the on-board Nextion route depends on a
   second harness that is not yet documented.

This audit records the discrepancy. It does **not** edit
[`s360-100-r4-core.md`](s360-100-r4-core.md) and does **not**
choose between the two possibilities.

### Per-fan-output drive topology (module side)

The module-side schematic places one low-side N-channel MOSFET
(`Q_NMOS_GSD` — `Q1..Q4`) per fan output, with a 1 kΩ gate
resistor (`R1`, `R2`, `R6`, `R7`) on the `TachPMW*` gate path.
Drain is tied to the shared `TachIO` net; source to `GND`. The
4-pin fan-output connectors (`J1`, `J2`, `J4`, `J5`) each carry
`+12V`, `GND`, `Pul_Cou*`, `TachPMW*`.

This is recorded as **observed schematic topology**, not as a
verified design intent. The schematic does not state:

- whether `TachPMW*` is the PWM drive signal or only the
  open-drain pull-down for a tach-share scheme (the net name is
  ambiguous on its own),
- whether the per-fan `Pul_Cou*` pin in the connector is a direct
  tach feedback from the fan that the module passes through to the
  Core, or whether some additional buffering / level shifting is
  present on the module side,
- the per-fan rated current, the MOSFET part number's actual
  ratings (the symbol is `Q_NMOS_GSD`, not a populated part
  number), or any thermal characterisation.

Bench / harness / waveform evidence is **owed** to the
HW-PINMAP-311 follow-up.

downstream package / product / WebFlash work must supply, at
minimum:

1. **Bench verification.** Populated `S360-311-R4` board on a
   `S360-100-R4` Core, with operator / reviewer / serial recorded,

   - the per-fan PWM drive waveform at the module-side fan-output

### Operator decisions (PWM-BLOCKER-REMOVAL-001, 2026-05-25)

| # | Question | Operator decision |
|---|---|---|
| D1 | Single-channel vs four-channel canonical FanPWM abstraction | **Four-channel.** Canonical FanPWM exposes **4 independent fan PWM outputs + 4 tach inputs**, matching the board outputs `J1` / `J2` / `J4` / `J5` and the legacy `sense360_fan_pwm.yaml` shape. Product layers may use one channel, gang channels, or leave channels unused, but the **package abstraction represents all four hardware channels neutrally**. |
| D2 | SX1509-routed vs direct-ESP32 PWM / tach binding | **SX1509 expander.** Bind `TachPMW1..4` → **SX1509 channels 0..3** and `Pul_Cou1..4` → **SX1509 channels 4..7**; keep `TachIO` **direct on ESP32 `IO16`**. Do **not** preserve the older direct-ESP32 binding where it contradicts the schematic. If implementation needs a Core abstract-bus / SX1509 package follow-up, **record that as the next prerequisite** rather than fabricating direct-GPIO mappings. |
| D3 | Is the committed JST-SH schematic the current R4, given Core `C02` → XH2.54? | **Committed schematic is current R4.** JST-SH is correct for the module side (`SM04B` fan outputs, `SM13B` From-Core `J3`). The Core ↔ module harness **adapts XH2.54 ↔ JST-SH** where needed. No newer schematic / BOM is asserted; no new schematic is needed for this PR. |

**What this resolves (design-intent layer only):**

- **D1 + D2** define the reconciliation target for
  [`packages/expansions/fan_pwm.yaml`](../../packages/expansions/fan_pwm.yaml):
  a **four-channel** package bound to **SX1509 channels 0..3 (PWM) /
  4..7 (tach)** with `TachIO` direct on `IO16`. This removes the
  cardinality and routing-direction ambiguity that previously gated
  rows 4 and 7. It does **not** remove the implementation gate: the
  parent Core abstract packages bind direct ESP32 GPIOs and
  `expansion_gpio1..4` were retired by `CORE-ABSTRACT-BUS-001C`, so the
  SX1509 four-channel binding is owed to a **`CORE-ABSTRACT-BUS-001` /
  SX1509 package follow-up** that must land before (or with)
  `PACKAGE-PWM-001-IMPLEMENT-001`.
- **D3** resolves the connector-identity sub-item raised in
  [§Project-tracker corroboration](#project-tracker-corroboration-drive-sense360_r4_tracker-evidence-pass-2):
  the committed JST-SH module schematic is current; the XH2.54-vs-JST-SH
  difference is a **harness adaptation**, not a stale module schematic.
  The Core `J6` / module `J3` 1-to-13 **silkscreen pin order** remains
  `verify` (no R4 silkscreen exists yet per tracker `G01`).

**What this does NOT resolve (still owed to bench / follow-up):** PWM
polarity (active-high vs active-low for the low-side N-FET gate path);

/ module `J3` silkscreen 1-to-13 pin order; the UART-on-`J3`-pins-11/12
routing question; and the `CORE-ABSTRACT-BUS-001` / SX1509 four-channel
package binding itself.

## Tach / RPM strategy (PACKAGE-PWM-TACH-STRATEGY-001)

This section is the **PACKAGE-PWM-TACH-STRATEGY-001** (2026-05-25)
decision record for how — and whether — the S360-311 / FanPWM lane
exposes fan tachometer / RPM data. It exists to **classify and resolve**
the tach/RPM blocker surfaced when `CORE-ABSTRACT-BUS-SX1509-001` landed
the neutral binding layer

(see [§Next-PR designation](#next-pr-designation) and the
[`CORE-ABSTRACT-BUS-SX1509-001` audit-log row](#hw-pinmap-311-followup-audit-log)).

This section is **documentation only**. It does **not** edit any
package, product, config JSON, WebFlash wrapper, workflow, script,
test, component, header, firmware, or manifest; it does **not** add a
FanPWM product / WebFlash / release surface, build a custom component,
promote `S360-311` `schematic_status`, or claim RPM support of any
kind. It records a scope decision and re-points the next implementation
PR.

### Finding being resolved (compile-proven — PWM-SX1509-TACH-PROOF-001)

`CORE-ABSTRACT-BUS-SX1509-001` first recorded this ESPHome capability
finding; **PWM-SX1509-TACH-PROOF-001 (2026-05-25) upgrades it from an
inferred claim to a compile/config proof** so the wording no longer
rests on "online docs" or on reasoning-from-`InternalGPIOPin` alone:

- **SX1509 PWM drive (channels 0..3, `TachPMW1..4`) IS supported** via
  the on-chip LED-driver PWM engine (`output: platform: sx1509`). This
  is **not** in question and stays the basis of FanPWM drive. SX1509
  GPIO / binary input (channels 4..7, polled `binary_sensor: platform:
  gpio`) is supported too.
- **An SX1509 expander pin used as an ESPHome `pulse_counter` input is
  compile-proven unsupported by ESPHome validation.** The minimal
  fixture
  [`tests/esphome/sx1509_pulse_counter_proof.yaml`](../../tests/esphome/sx1509_pulse_counter_proof.yaml)
  (SX1509 hub on `core_i2c`, channel 4 = `Pul_Cou1`, behind a
  `sensor: platform: pulse_counter`) is **rejected by `esphome config`**
  (ESPHome 2026.5.1, exit code 2) with the captured error:

  ```text
  Failed config

  sensor.pulse_counter: [source sx1509_pulse_counter_proof.yaml:…]
    pin:
      [sx1509] is an invalid option for [pin]. Please check the indentation.
  ```

  The `pulse_counter` pin schema accepts only an interrupt-capable
  native ESP32 `InternalGPIOPin`; the SX1509 expander pin schema is not
  a valid option for it.
- **Per-fan RPM via `Pul_Cou1..4` is therefore un-implementable** as a
  standard ESPHome `pulse_counter` RPM input — a compile-proven blocker
  for the originally-scoped four-PWM + four-per-fan-RPM
  `PACKAGE-PWM-001-IMPLEMENT-001`.

Two control checks in
[`tests/test_sx1509_tach_pulse_counter_proof.py`](../../tests/test_sx1509_tach_pulse_counter_proof.py)
confirm the rejection is specific to `pulse_counter` + SX1509, not a
generic config error: the **same** SX1509 pin validates as a
`binary_sensor: platform: gpio` (`esphome config` passes), and
`pulse_counter` validates on a **native** ESP32 GPIO (the `TachIO` /
`GPIO16` line) (`esphome config` passes). The decision below
(PWM-drive-only first) is unchanged by the proof — it confirms, rather
than revises, `PACKAGE-PWM-TACH-STRATEGY-001`.

Cross-checks consistent with the finding (not new evidence): the only
`pulse_counter` uses in the repo are on **native ESP32 GPIOs**
([`fan_pwm.yaml`](../../packages/expansions/fan_pwm.yaml) `${fan_tach_pin}`,
[`sense360_fan_pwm.yaml`](../../packages/expansions/sense360_fan_pwm.yaml)
`GPIO8`/`GPIO12`/`GPIO14`/`GPIO16`,
[`sense360_core_mapping.yaml`](../../packages/hardware/sense360_core_mapping.yaml)
`GPIO5`); both SX1509 packages
([`fan_pwm_sx1509.yaml`](../../packages/expansions/fan_pwm_sx1509.yaml),
[`gpio_expander_sx1509.yaml`](../../packages/expansions/gpio_expander_sx1509.yaml))
expose tach as `binary_sensor: platform: gpio`, never `pulse_counter`;
no custom tach/RPM component exists in [`components/`](../../components).
