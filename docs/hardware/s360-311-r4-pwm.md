# S360-311-R4 PWM Fan Driver Pin / Package Mapping Audit (HW-PINMAP-311)

## Status

**Status: partial — schematic evidence available; package reconciliation pending.**

This document is the **HW-PINMAP-311** audit record for the
`Sense360 PWM` board, revision **R4** (`S360-311-R4`). It exists to
close the documentation gap recorded at
[`docs/hardware/hardware-artifact-policy.md` Follow-up PR sequence](hardware-artifact-policy.md#follow-up-pr-sequence)
and at
[`docs/hardware/board-readiness-matrix.md` Follow-up PR sequence #2](board-readiness-matrix.md#follow-up-pr-sequence),
which together call for a per-board pin/package mapping audit
**before** any `FanPWM`-bearing package, product, build, or
WebFlash work can proceed.

Unlike HW-PINMAP-310 (which is `pending — schematic/design evidence
required`), HW-PINMAP-311 has a committed module-side schematic to
work from: the curated artifact index
[`docs/hardware/artifacts/S360-311-R4.md`](artifacts/S360-311-R4.md)
landed under HW-ASSETS-003 with the schematic PDF at
[`docs/hardware/schematics/S360-311-R4.pdf`](schematics/S360-311-R4.pdf).
This audit consumes that evidence to record a schematic-backed
reconciliation view. It does **not** resolve the firmware-side
disagreements it surfaces — those are owed to separate,
evidence-bearing follow-up PRs.

## Schematic summary

This section restates only what is directly visible in the
committed schematic PDF
[`docs/hardware/schematics/S360-311-R4.pdf`](schematics/S360-311-R4.pdf)
and inventoried in
[`docs/hardware/artifacts/S360-311-R4.md` What the schematic appears to contain](artifacts/S360-311-R4.md#what-the-schematic-appears-to-contain).
It does **not** invent electrical characteristics, transistor
topology, timing guarantees, MOSFET specs, or pull-up assumptions
beyond what is labelled on the sheet.

- **Voltage Boost** section. `MT3608` boost converter (`U1`) with
  input inductor `L1 22 µH`, Schottky `D1 SS34`, feedback divider
  `R3 38 kΩ` / `R5 2 kΩ`, input cap `C1 22 µF / 10 V`, output cap
  `C2 22 µF / 10 V`, output indicator `R4 330 Ω` + blue LED `D2`.
  Boosts the Core-supplied `+5V` rail up to `+12V` for the fan
  outputs.
- **Fan outputs section** (labelled `"NINE 4pin FANs"` in the sheet
  title; see [§Open documentation questions](#open-documentation-questions)).
  **Four** 4-pin fan output connectors are present: `J1`, `J2`,
  `J4`, `J5`. Each fan connector has pins `+12V`, `GND`,
  `Pul_Cou*`, `TachPMW*` and is paired with one low-side N-channel
  MOSFET (`Q_NMOS_GSD` — `Q1..Q4`) on the `TachPMW*` gate path,
  with a 1 kΩ gate resistor (`R1`, `R2`, `R6`, `R7`); drain wired
  to the shared `TachIO` net; source to `GND`.
- **"From Core"** module-side connector `J3` — 13-pin. Pin / net
  table as labelled in the PDF:

  | Module-side `J3` pin | Net |
  |---:|---|
  | 1 | `+5V` |
  | 2 | `TachIO` |
  | 3 | `TachPMW1` |
  | 4 | `Pul_Cou1` |
  | 5 | `TachPMW2` |
  | 6 | `Pul_Cou2` |
  | 7 | `TachPMW3` |
  | 8 | `Pul_Cou3` |
  | 9 | `TachPMW4` |
  | 10 | `Pul_Cou4` |
  | 11 | `UART_RX` |
  | 12 | `UART_TX` |
  | 13 | `GND` |

- **"NEXTION DISPLAY"** connector `J6` (module-side) — 4-pin:
  `+5V`, `ESP32_TXD` / `UART_TX` (to display RX), `ESP32_RXD` /
  `UART_RX` (to display TX), `GND`. On-board route from module
  `J3` pins 11 / 12 to module `J6` UART pair.
- **Mounting holes** `H1`, `H2`, `H3`, `H4`.

Everything beyond the list above — module dimensions, exact
component values not printed on the visible sheet, on-board PCB
silkscreen, layer stackup, fab notes, harness identity, or any

from the retained-but-not-committed KiCad source / KiCad PCB / BOM

## Connector / pin-map findings

### Module-side `J3` ↔ Core-side `J6` (13-pin harness)

The two connectors mate via a 13-pin Core ↔ module harness. The
module-side capture (above) is **schematic-confirmed** against the
committed S360-311-R4 PDF. The Core-side capture in
[`s360-100-r4-core.md` §J6](s360-100-r4-core.md#j6--12-v-pwm-fan-connector-13-pin)
records the net list but explicitly marks the 1-to-13 pin order as
**verify** against the silkscreen (
[`s360-100-r4-core.md` Open Questions #9](s360-100-r4-core.md#open-questions--verification-needed)).

| Element | Core side (`J6`) | Module side (`J3`) | Reconciliation status |
|---|---|---|---|
| Connector identity | `J6` — 12 V PWM fan connector (13-pin) — [§J6](s360-100-r4-core.md#j6--12-v-pwm-fan-connector-13-pin) | `J3` — "From Core" (13-pin) — schematic-confirmed | Naming mismatch is expected (each board labels its own connectors); 13-pin pin count matches. |
| Net list (signal nets only) | `+5V`, `GND`, `TachIO`, `TachPMW1..4`, `Pul_Cou1..4` | `+5V`, `GND`, `TachIO`, `TachPMW1..4`, `Pul_Cou1..4`, **`UART_RX`**, **`UART_TX`** | **Module side carries two additional nets (`UART_RX` / `UART_TX`) that the Core-side capture does not currently document.** See [§UART pins on `J3` pins 11–12](#uart-pins-on-j3-pins-11-12). |

### UART pins on `J3` pins 11–12

The module-side `J3` labels pins 11 / 12 as `UART_RX` / `UART_TX`
and routes them on-board to the 4-pin Nextion display connector
`J6` (module side). The Core-side `J6` capture in
[`s360-100-r4-core.md` §J6](s360-100-r4-core.md#j6--12-v-pwm-fan-connector-13-pin)
does **not** currently list any UART pins on the 13-pin connector
— only `+5V`, `GND`, `TachIO`, and the eight `TachPMW*` /
`Pul_Cou*` signals (11 nets total) are recorded. Two
schematic-internally-consistent possibilities remain, **both in
scope for the HW-PINMAP-311 follow-up**:

1. **The Core-side `J6` capture is incomplete.** Pins 11 / 12 on
   the Core-side `J6` connector physically carry the UART pair
   (e.g. from `TXD0` / `RXD0` or a different ESP32 UART), and the
   capture in
   [`s360-100-r4-core.md` §J6](s360-100-r4-core.md#j6--12-v-pwm-fan-connector-13-pin)
   must be extended to record those two nets.
2. **The Nextion UART pair is delivered to the module via a
   separate harness, not via the 13-pin Core `J6` cable.** Pins
   11 / 12 on module-side `J3` would then be expected to be left
   unconnected (or to host different signals) on the actual
   13-pin cable. In this case the module-side schematic labels
   are misleading and the on-board Nextion route depends on a
   second harness that is not yet documented.

This audit records the discrepancy. It does **not** edit
[`s360-100-r4-core.md`](s360-100-r4-core.md) and does **not**
choose between the two possibilities.

### Per-fan-output drive topology (module side)

The module-side schematic places one low-side N-channel MOSFET
(`Q_NMOS_GSD` — `Q1..Q4`) per fan output, with a 1 kΩ gate
resistor (`R1`, `R2`, `R6`, `R7`) on the `TachPMW*` gate path.
Drain is tied to the shared `TachIO` net; source to `GND`. The
4-pin fan-output connectors (`J1`, `J2`, `J4`, `J5`) each carry
`+12V`, `GND`, `Pul_Cou*`, `TachPMW*`.

This is recorded as **observed schematic topology**, not as a
verified design intent. The schematic does not state:

- whether `TachPMW*` is the PWM drive signal or only the
  open-drain pull-down for a tach-share scheme (the net name is
  ambiguous on its own),
- whether the per-fan `Pul_Cou*` pin in the connector is a direct
  tach feedback from the fan that the module passes through to the
  Core, or whether some additional buffering / level shifting is
  present on the module side,
- the per-fan rated current, the MOSFET part number's actual
  ratings (the symbol is `Q_NMOS_GSD`, not a populated part
  number), or any thermal characterisation.

Bench / harness / waveform evidence is **owed** to the
HW-PINMAP-311 follow-up.
