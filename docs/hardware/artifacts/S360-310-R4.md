# S360-310-R4 Hardware Artifact Index (HW-ASSETS-310)

## Purpose and scope

This document is the **curated artifact index** for the Sense360 Relay
board, revision R4 (`S360-310-R4`). It records the schematic evidence
delivered under HW-ASSETS-310, classifies it against the
[Hardware Artifact Policy](../hardware-artifact-policy.md), and
inventories what is still missing before pin-map, package, product, or
WebFlash work can proceed.

### What the schematic appears to contain

Observed directly from the uploaded PDF. Every claim below is sourced
from a visible label or symbol on the single schematic sheet.

- **Section title: "INLINE FAN CONTROL USING RELAY".** One section,
  no sub-sheets.
- **"From Core" connector `J2`** — 3-pin. Net list as labelled in the
  PDF:

  | Module J2 pin | Net |
  |---:|---|
  | 1 | `+5V` |
  | 2 | `Relay` |
  | 3 | `GND` |

  The net order (`+5V` → `Relay` → `GND`) matches the Core-side `J4`
  capture in
  [`s360-100-r4-core.md` §J4](../s360-100-r4-core.md#j4--relay-module-connector-3-pin)
  pin-for-pin by net. Silkscreen pin-1 location on the module side is
  **not** derivable from the PDF and remains owed to
  `HW-PINMAP-310-FOLLOWUP`.
- **Relay coil-drive topology.** Low-side NPN BJT switch from a
  module-supplied `+5V` rail (no on-board boost):
  - `Q1 MMBT3904` — NPN BJT in SOT-23. Emitter to `GND`, collector to
    the relay coil node.
  - `R1 1 kΩ` — base series resistor on the `Relay` signal path from
    `J2` pin 2 to the `Q1` base node.
  - `R2 10 kΩ` — base-to-`GND` pull-down at the `Q1` base node
    (keeps `Q1` off when the `Relay` net is floating or Hi-Z).
  - `D1` — flyback diode across the relay coil (cathode to `+5V`,
    anode to the `Q1` collector / coil node). No part number visible.
- **Relay `K1`.** Mechanical relay. Coil between `+5V` and `Q1`
  collector; flyback `D1` across the coil as above. **No relay part
  number, no coil-voltage label, and no contact-current rating
  visible on the schematic sheet.** The relay reference is `K1` only.
- **"Inline Fan" load-side connector `J1`** — 3-pin. Wired to the
  switching-contact side of `K1` (visible contact-side pin labels
  `1` / `2` / `3`). A 3-pin output is consistent with SPDT contact
  topology (`NO` / `COM` / `NC`), but the specific NO / COM / NC
  assignment to `J1` pins 1 / 2 / 3 is **not** labelled on the
  schematic and is owed to silkscreen evidence under
  `HW-PINMAP-310-FOLLOWUP`.
- **What the schematic does NOT visibly contain.**
  - **No opto-isolator** between the `Relay` input and the relay coil
    (the `Q1` MMBT3904 is a direct low-side switch).
  - **No on-board indicator LED** (no LED reference is present on
    this sheet; no `R*` current-limit resistor labelled for an
    indicator).
  - **No mains-side snubber** (no RC snubber or MOV across the `K1`
    switching contacts on the `J1` side).
  - **No "NEXTION DISPLAY" connector** (unlike `S360-311-R4` /
    `S360-312-R4`).
  - **No mounting holes** (`H1`..`H4`) are referenced on this sheet.
  - **No I²C / UART / DAC / boost converter** circuitry.

### Reconciliation flags raised or strengthened by this schematic

1. **Module `J2` vs Core `J4` connector net order (confirmed by net,
   silkscreen still owed).** Module `J2` carries
   `+5V` / `Relay` / `GND` on pins 1 / 2 / 3, matching the
   Core-side `J4` capture in
   [`s360-100-r4-core.md` §J4](../s360-100-r4-core.md#j4--relay-module-connector-3-pin)
   pin-for-pin by net. **Silkscreen pin-1 location on the module
   side is not derivable from the PDF and remains owed to
   `HW-PINMAP-310-FOLLOWUP`.**
2. **`Relay` net drive level (consistent with direct ESP32 `IO3`).**
   The `Relay` net enters the module at `J2` pin 2, feeds the `Q1`
   MMBT3904 base via `R1` 1 kΩ, with `R2` 10 kΩ as base pull-down.
   No level shifter is present on the module's `Relay` input;
   the topology is consistent with direct 3.3 V CMOS drive from the
   Core's `IO3` net captured at
   [`s360-100-r4-core.md` Pin assignments](../s360-100-r4-core.md#pin-assignments)
   line 100. This does **not** resolve the abstract-bus disagreement
   (see flag #6 below).
3. **Coil rail (confirmed `+5V`, no on-board boost).** The relay
   coil is driven from the `+5V` rail at `J2` pin 1. There is **no**
   MT3608 / boost converter on the sheet; the Core-supplied `+5V` is
   the coil rail. This confirms the assumption recorded in
   [`s360-310-r4-relay.md` Known assumptions](../s360-310-r4-relay.md#known-assumptions)
   bullet "Power rail at `J4` pin 1 is `+5V`".
4. **No opto-isolation (new flag).** There is no opto-isolator
   between the `Relay` input and the relay coil drive transistor.
   The relay's coil-side and contact-side share the same module
   ground reference. Mains-side isolation is therefore the relay's
   contact-to-coil isolation alone (relay part identity not
   visible — see flag #7). This is **not** a mains-voltage
   compliance claim; mains-voltage compliance for any
   `FanRelay`-bearing config that switches mains loads is owed by a
   separate COMPLIANCE-001-equivalent slice if and when such a
   product surface is proposed.
5. **No on-board indicator LED (new flag).** No indicator LED
   appears on the schematic for relay-state visibility. Whether the
   module should carry an indicator is a board-design decision, not
   a firmware question.
6. **`IO3` (Core schematic) vs `GPIO4` (ceiling Core abstract
   `relay_pin`) vs `GPIO10` (generic Core abstract `relay_pin`)
   disagreement is NOT resolved by this schematic.** The module
   schematic confirms only that the module receives a `Relay` net on
   `J2` pin 2 and drives a 5 V coil via `Q1` MMBT3904. It does
   **not** establish which Core-side GPIO physically connects to
   that net, because the Core schematic and the Core abstract
   packages disagree and the module sheet does not capture the Core
   side. Resolution remains owned by
   [`s360-310-r4-relay.md` Pin-map reconciliation status](../s360-310-r4-relay.md#pin-map-reconciliation-status),
   `HW-PINMAP-310-FOLLOWUP`, and `CORE-ABSTRACT-BUS-001` per
   [`release-one-hardware-audit.md` Required follow-ups #2 / #3](../../release-one-hardware-audit.md#required-follow-ups).
   This index records the disagreement; it does **not** edit
   [`packages/hardware/sense360_core_ceiling.yaml`](../../../packages/hardware/sense360_core_ceiling.yaml),
   [`packages/hardware/sense360_core.yaml`](../../../packages/hardware/sense360_core.yaml),
   [`packages/expansions/fan_relay.yaml`](../../../packages/expansions/fan_relay.yaml),
   or [`s360-100-r4-core.md`](../s360-100-r4-core.md).
7. **Relay part identity, coil voltage label, and contact rating
   missing from sheet (new flag).** The relay reference is `K1`
   only; no part number, no coil-voltage label, no contact-current
   rating. The hardware-catalog description
   (`On/off relay for bathroom fans.`) is consistent with a small
   mechanical relay, but the contact rating that determines which
   fan loads are safe to switch is **not** schematic-visible and
   must come from the BOM.
8. **`J1` `NO` / `COM` / `NC` pin order is silkscreen-only.** `J1`
   is a 3-pin contact-side output. Whether pins 1 / 2 / 3 map to
   `NO` / `COM` / `NC`, or some other ordering, is **not** labelled
   on the schematic and must come from the silkscreen.

This index does not edit
[`docs/hardware/firmware-package-mapping-audit.md`](../firmware-package-mapping-audit.md)
or any package YAML.
