# S360-312-R4 Hardware Artifact Index (HW-ASSETS-003)

## Purpose and scope

This document is the **curated artifact index** for the Sense360 DAC
(0–10 V analog) fan board, revision R4 (`S360-312-R4`). It records the
schematic evidence delivered under HW-ASSETS-003, classifies it
against the
[Hardware Artifact Policy (archived)](../../archive-index.md), and
inventories what is still missing before pin-map, package, product, or
WebFlash work can proceed.

This index is **documentation only**. It does not:

- change any value in [`config/hardware-catalog.json`](../../../config/hardware-catalog.json),
  [`config/product-catalog.json`](../../../config/product-catalog.json),
  [`config/webflash-builds.json`](../../../config/webflash-builds.json),
  or [`config/webflash-compatibility.json`](../../../config/webflash-compatibility.json),
- change any product YAML under [`products/`](../../../products/),
  any WebFlash wrapper under
  [`products/webflash/`](../../../products/webflash/), or any package
  YAML under [`packages/`](../../../packages/),
- change any workflow under `.github/workflows/`, any script under
  `scripts/`, any test under `tests/`, any component under
  `components/`, any header under `include/`, or `.gitignore`,
- mark `S360-312-R4` `verified` — `schematic_status` stays
  `cataloged_unverified`,
- add or change `schematic_file` for `S360-312` in
  [`config/hardware-catalog.json`](../../../config/hardware-catalog.json),
- promote `S360-312` to `preview` / `stable` / `production`,
- add a `FanDAC`-bearing entry to
  [`config/webflash-builds.json`](../../../config/webflash-builds.json)
  or to any product / WebFlash wrapper,
- relax or change the explicit `FanDAC` ↔ `AirIQ` conflict in
  [`config/webflash-compatibility.json`](../../../config/webflash-compatibility.json)
  (`rules.fandac_conflicts_with_airiq: true`),
- regenerate firmware, create a release, or change any WebFlash
  manifest.

**Schematic evidence uploaded ≠ schematic verified.** Promotion from
`cataloged_unverified` to `verified` requires the `HW-PINMAP-312`
follow-up (pin-map reconciliation against the Core J7 capture
including the **voltage-rail discrepancy** flagged below, the I²C
address-selection scheme on the two GP8403 DACs, and the package YAML
[`packages/expansions/fan_gp8403.yaml`](../../../packages/expansions/fan_gp8403.yaml))
plus a separate JSON catalog change.

The Release-One shipping configuration remains
`Ceiling-POE-VentIQ-RoomIQ` with artifact
`Sense360-Ceiling-POE-VentIQ-RoomIQ-v1.0.0-stable.bin`. The LED
preview path (`Ceiling-POE-VentIQ-RoomIQ-LED`) is unchanged. FanDAC
is not, and is not made by this index, part of any current Release-One
or WebFlash build.

### What the schematic appears to contain

Observed directly from the uploaded PDF. Every claim below is sourced
from a visible label or symbol on the single schematic sheet.

- **Voltage Boost** section. `MT3608` boost converter (`U1`), input
  inductor `L1 22µH`, Schottky `D1 SS34`, feedback divider `R1 2k` /
  `R2 38k`, input cap `C1 22µF/0805/10V`, output cap `C2 22µF/0805/10V`,
  output indicator `R9 500R` + blue LED `D6`. Input rail at the boost
  converter `IN` pin is labelled `+3.3V`; output rail is `+12V`. (The
  input-rail value is one of the reconciliation flags below — Core J7
  is captured as `+5V` on pin 1.)
- **Two GP8403 DAC channels** on a shared I²C bus:
  - **"CLOUDLIFT S12 FAN"** — `IC1 GP8403-TC50-EW` (10-pin TSSOP +
    exposed pad `EP`). Pins: `SCLK` (1) ← `I2C_SCL`, `SDA` (2) ←
    `I2C_SDA`, `A0` (3), `A1` (4), `VCC` (5), `GND` (6), `VOUT1` (7),
    `VOUT0` (8), `A2` (9), `V5V` (10), `EP` (11) → `GND`. Address pins
    `A0`, `A1`, `A2` are selectable via a 6-position DIP `SW1` with
    4.7 kΩ pullups `R3`, `R5`, `R7`. `VOUT0` → `C7 10µF` + `D4` blocking
    diode → `J2` connector. `VOUT1` → `C8 10µF` + `D2` blocking diode →
    `J2` connector. Output decoupling cap `C5 1µF`.
  - **"CLOUDLIFT S12 FAN2"** — `IC2 GP8403-TC50-EW`, mirror of `IC1`.
    Address pins are labelled `2A0`, `2A1`, `2A2` (distinct address
    selectable via DIP `SW2` with pullups `R4`, `R6`, `R8`). `VOUT0` →
    `C9 10µF` + `D5` blocking diode → `J3` connector. `VOUT1` →
    `C10 10µF` + `D3` blocking diode → `J3` connector. Output decoupling
    cap `C6 1µF`.
- **Cloudlift S12 fan output connectors** `J2` and `J3` — each 3-pin
  (`GND` / `vout0` / `vout1` net labels; precise silkscreen pin order
  to be re-confirmed under `HW-PINMAP-312`).
- **DAC bypass caps.** `C3 100n` (`IC1` VCC) and `C4 100n` (`IC2` VCC).
- **"From Core"** connector `J1` — 6-pin. Net list as labelled in the
  PDF:

  | Module J1 pin | Net |
  |---:|---|
  | 1 | `+3.3V` |
  | 2 | `I2C_SDA` |
  | 3 | `I2C_SCL` |
  | 4 | `UART_RX` |
  | 5 | `UART_TX` |
  | 6 | `GND` |

- **"NEXTION DISPLAY"** connector `J7` — 4-pin: `+5V`,
  `ESP32_RXD` / `UART_RX` (to screen TX), `ESP32_TXD` / `UART_TX`
  (to screen RX), `GND`.
- **Mounting holes** `H1`, `H2`, `H3`, `H4`.

### Reconciliation flags raised or strengthened by this schematic

1. **Core J7 vs Module J1 voltage rail (new flag).**
   [`s360-100-r4-core.md` §J7](../s360-100-r4-core.md#j7--gp8403-fan-connector-6-pin)
   captures the Core J7 pin-1 supply as `+5V`. The module-side `J1`
   labels pin 1 as `+3.3V`. Either:
   - the Core J7 capture is incorrect and the supply is actually
     `+3.3V`, or
   - the module-side schematic mis-labels the rail and the board
     actually accepts `+5V` (consistent with boosting to `+12V` via
     MT3608 at a more comfortable ratio), or
   - the Core and module agree only by accident on pin position and
     the rail differs on the silkscreen.

   **This is a wiring-relevant flag and must be resolved against
   physical silkscreen evidence before any FanDAC-bearing config
   ships.** Belongs to `HW-PINMAP-312`. This index does not edit
   [`s360-100-r4-core.md`](../s360-100-r4-core.md),
   [`packages/expansions/fan_gp8403.yaml`](../../../packages/expansions/fan_gp8403.yaml),
   or any other artifact; it only records the discrepancy.
2. **Dual GP8403 / dual Cloudlift-style output (new info).** The
   hardware-catalog description "0 to 10V analog fan driver, for
   example Cloudlift S12" is singular, but the schematic clearly
   carries **two** `GP8403-TC50-EW` DACs (`IC1` and `IC2`) on a shared
   I²C bus, each addressable via its own DIP switch (`SW1` / `SW2`),
   each driving its own 3-pin Cloudlift-style output (`J2` / `J3`).
   This expands the board's intended capacity beyond what the catalog
   description currently communicates. **Documentation question
   only**; no behaviour change. To be resolved against silkscreen and
   reflected in the standalone reference doc under `HW-PINMAP-312`.
3. **I²C address-selection scheme (new info).** The two DACs each have
   three address pins (`A0`/`A1`/`A2` on `IC1`; `2A0`/`2A1`/`2A2` on
   `IC2`) routed through 4.7 kΩ pullups (`R3`/`R5`/`R7` for `IC1`;
   `R4`/`R6`/`R8` for `IC2`) to a 6-position DIP switch (`SW1` /
   `SW2`) tied to `GND`. Any future
   [`packages/expansions/fan_gp8403.yaml`](../../../packages/expansions/fan_gp8403.yaml)
   reconciliation must specify the operator-chosen I²C address(es) —
   the package YAML cannot assume a single address.
4. **Shared I²C bus (pre-existing, confirmed).** Module pin 2 / pin 3
   `I2C_SDA` / `I2C_SCL` map onto the Core's shared I²C bus
   ([`s360-100-r4-core.md` I2C bus](../s360-100-r4-core.md#i2c-bus)).
   Address-collision review against the Core's existing I²C devices
   belongs to `HW-PINMAP-312`.
5. **Nextion UART connector (new info).** A dedicated 4-pin
   `NEXTION DISPLAY` connector `J7` on the module side carries the
   ESP32 UART pair. The Core J7 6-pin capture's pin-4 / pin-5
   `UART_RX` / `UART_TX` lines reach this Nextion connector.

or any package YAML.

   Pre-requisite for any FanDAC productization.

4. **Cloudlift S12 output connector pin order.** `J2` / `J3` are

   Same open as for S360-100-R4 — see
