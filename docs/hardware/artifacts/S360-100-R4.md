# S360-100-R4 Hardware Artifact Index (HW-ASSETS-002)

## Purpose and scope

This document is the **curated artifact index** for the Sense360 Core
main board, revision R4 (`S360-100-R4`). It inventories the hardware
artifacts that have been delivered to this repository's task
environment under HW-ASSETS-002, classifies them against the
[Hardware Artifact Policy](../hardware-artifact-policy.md), and
records what is committed, what is excluded, and what remains open.

This index is **documentation only**. It does not:

- change any value in `config/hardware-catalog.json`,
  `config/product-catalog.json`, `config/webflash-builds.json`, or
  `config/webflash-compatibility.json`,
- change any product YAML under `products/**`, any WebFlash wrapper
  under `products/webflash/**`, or any package YAML under
  `packages/**`,
- change any workflow, script, test, component, header, example
  file, or `.gitignore`,
- commit any raw `.zip` archive, KiCad source, KiCad PCB, KiCad
  project metadata, BOM, CPL / pick-and-place CSV, gerbers, drill
  files, STEP model, or board images,
- regenerate firmware, create a GitHub Release, or change any
  WebFlash manifest,
- unblock FanTRIAC (HW-005 stays a separate gate),
- change Sense360 LED's exclusion from production Release-One.

The Release-One shipping configuration remains
`Ceiling-POE-VentIQ-RoomIQ` with artifact
`Sense360-Ceiling-POE-VentIQ-RoomIQ-v1.0.0-stable.bin`. The LED
preview path (`Ceiling-POE-VentIQ-RoomIQ-LED`) is unchanged.

## Policy reference

This index applies
[`docs/hardware/hardware-artifact-policy.md`](../hardware-artifact-policy.md)
(HW-ASSETS-001). The relevant rules:

- No raw, unreviewed ZIP / archive is committed as an opaque blob.
- Mac OS metadata, editor / IDE history, embedded `.git/`, vendor
  scratch exports, KiCad temporary / cache / backup files, and
  private supplier / customer data are excluded outright.
- KiCad source, KiCad PCB, KiCad project metadata, BOM, CPL,
  gerbers, drill files, STEP, and board images are
  **retained-but-not-committed** under the current policy.
- Per-board indexes record file sizes and SHA256 for every artifact
  actually present in the upload. Missing files are marked
  `not provided in this upload`; checksums are never fabricated.
- Hardware artifact availability does **not** imply firmware
  support, WebFlash availability, build-matrix inclusion,
  product-catalog status, or Release-One promotion.
- No Git LFS is introduced. A long-term storage decision for
  retained-but-not-committed artifacts is **open**.

## Relationship to firmware package mapping

[`docs/hardware/firmware-package-mapping-audit.md`](../firmware-package-mapping-audit.md)
(HW-009) audits firmware-vs-schematic mapping for each package YAML
referenced by the Release-One stack. The S360-100-R4 evidence
recorded here supports — but does not change — the following
reconciliation flags already tracked there and in
[`docs/hardware/s360-100-r4-core.md`](../s360-100-r4-core.md):

- **`LED_DATA` / `IO38`.** Core `IO38 = LED_DATA` → `U2A` 74LVC1G07
  buffer → `R8` 330 Ω → `J3` → S360-300 `J1`. Already resolved at the
  package level for the ceiling LED package
  ([`packages/hardware/led_ring_ceiling.yaml`](../../../packages/hardware/led_ring_ceiling.yaml),
  `led_data_pin: GPIO38`) under HW-010. Sense360 LED stays excluded
  from production Release-One.
- **FanTRIAC HW-005.** Core `IO5 = SEN0609_TX` and `IO6 =
  out(gpio6)`, both wired to the RoomIQ J10 connector. The
  `TRI_GPIO1` / `TRI_GPIO2` nets on `J15` appear only on the SX1509
  (U3) side of the Core sheet; `S360-320` schematic is not
  committed. FanTRIAC stays `blocked`.
- **Core J10 vs RoomIQ J6 pin-order.** Two `verified` schematics
  disagree on the 12-pin mating connector; resolution requires
  physical silkscreen inspection. Unchanged by this index.
- **`AirQ_Led` / `AirQ_Status_Led` net reuse on J9.** Open Question
  #4 in [`s360-100-r4-core.md`](../s360-100-r4-core.md#open-questions--verification-needed).
  Unchanged by this index.

The connector / pin / net data in the uploaded artifacts (the CPL
CSV's `Ref` / `Val` / `Package` rows for `J2`–`J15`, `U1`–`U6`,
`SW3`–`SW4`, `Q2` / `Q4`, `H1`–`H3`, and the test-point set
`TP1`–`TP13`) is consistent with the Core hardware reference doc and
is **not** independent re-verification of the open questions above.
This index does not edit
[`docs/hardware/firmware-package-mapping-audit.md`](../firmware-package-mapping-audit.md)
or any package YAML.

## Open questions / verification needed

Pre-existing Core open questions are tracked authoritatively in
[`docs/hardware/s360-100-r4-core.md`](../s360-100-r4-core.md#open-questions--verification-needed)
and are not re-litigated here. The items below are the
**artifact-specific** opens raised by this index:

1. **Long-term storage for retained-but-not-committed artifacts.**
   The BOM, CPL, gerbers, and STEP files for S360-100-R4 currently
   live only in the HW-ASSETS-002 task upload. Where should they
   live for the long term — a separate hardware-only repo, Git LFS
   in this repo, GitHub Releases per board / revision, or a supplier
   archive? Tracked at the policy level under
   [Future storage decision](../hardware-artifact-policy.md#future-storage-decision).
2. **KiCad source delivery.** No KiCad schematic / PCB / project
   metadata was uploaded under HW-ASSETS-002. Should a follow-up
   provide them? If so, the policy-level
   [Future storage decision](../hardware-artifact-policy.md#future-storage-decision)
   must be made first.
3. **Gerber inner-filename prefix.** The uploaded gerbers ZIP uses
   the legacy KiCad project name `Sense360_Core_R4-*` for inner
   layer files, not the canonical SKU prefix `S360-100-R4-*`. Should
   the source KiCad project be renamed at the project level so that
   future gerber exports use the canonical prefix? This is a hardware
   / KiCad-project question, not a firmware question.
4. **Image / render set.** No board images were uploaded. Should a
   curated, compressed set be added under
   [`docs/hardware/images/`](../images/) (a directory that does not
   yet exist) by a later PR for documentation purposes? Out of scope
   here.
5. **Manufacturing-archive retention policy.** Per-board manufacturing
   archives (BOM + CPL + gerbers + STEP) imply a retention contract.
   Until the future storage decision is made, this index records
   only inventory and checksum, not retention guarantees.

The pre-existing Core schematic-level open questions in
[`s360-100-r4-core.md`](../s360-100-r4-core.md#open-questions--verification-needed)
— the J6 / J10 pin-order discrepancy, the `IO10` net label, the
`IO39`–`IO42` ↔ TPx mapping, the J2 PoE harness identity, the
`AirQ_Led` / `AirQ_Status_Led` reuse, the FanTRIAC GPIO mapping —
remain owned there. This index does not edit them and does not add
new pin-level claims.
