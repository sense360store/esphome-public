## Known assumptions

Stated as assumptions, not as facts. None of these is sufficient
to mark `S360-400` `verified` or to mark `power_240v.yaml`
`confirmed-ok`, and none clears COMPLIANCE-001.

- **The module converts mains AC to a low-voltage DC rail
  suitable for the Core.** The catalog description records `Mains
  to 5V using HLK-5M05.`. The package header records `100-240V
  AC` input and `5V DC, 2A (10W)` output. Both strings imply a
  small isolated AC-DC converter; neither is schematic-evidenced
  for `S360-400-R4`.
- **The AC-DC converter is an HLK-family module.** Catalog says
  `HLK-5M05`; the package header says `HLK-PM01 or similar`.
  Which exact part is on `S360-400-R4` is **unknown** without the
  module-side schematic and the module BOM.
- **The module is off-board relative to Core.** The Core
  schematic enumerates `J2 = PoE_ACDC` (PoE inlet) and a USB-C
  `VBUS` inlet at `J8`; it does **not** enumerate a separate
  mains-input header for `S360-400`. The four `legacy-compatible`
  `*-pwr` products treat the 240 V PSU as a logical power option,
  not as a Core-mating connector.
- **Output rail merges into the Core `+5V` net.** The Core
  schematic shows `+5V` produced by an ideal-diode-OR network on
  `Q2 Si2319CDS` + `D12` between `VBUS` (USB-C) and `PoE_ACDC`
  (`s360-100-r4-core.md` Â§Power inputs and rails). Whether a
  `S360-400`-fed 5 V output joins that `+5V` rail through the
  same OR network, through `VBUS`, or through a distinct path is
  **unknown** without the schematic.

These assumptions are not invented rail maps. Anything beyond the
list above â€” mains-input topology (fuse, MOV, X/Y caps, common-mode
choke), transformer / opto isolation, creepage / clearance
distances, EMC behaviour, in-rush, output protection (OVP / OCP /
OTP), thermal derating, on-board indicators, harness identity,
country-specific certifications, enclosure / IP-rating
expectations â€” is **unknown** and must come from the module-side
schematic plus the COMPLIANCE-001 evidence checklist.
