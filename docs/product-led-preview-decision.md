# LED Preview Product Path Decision (PRODUCT-005)

## Purpose & scope

This document records the decision about whether and how the **Sense360
LED** board (`S360-300`) should get a future preview product path in this
repository, given the schematic evidence committed under HW-007, the
machine-readable status refresh in HW-008, the firmware-package mapping
audit in HW-009, and the ceiling-LED package pin fix in HW-010.

This document is **documentation only**. It does not:

- add, remove, or modify any product YAML, WebFlash wrapper, or package
  YAML,
- add, remove, or modify any entry in
  [`config/product-catalog.json`](../config/product-catalog.json),
  [`config/webflash-builds.json`](../config/webflash-builds.json),
  [`config/webflash-compatibility.json`](../config/webflash-compatibility.json),
  or [`config/hardware-catalog.json`](../config/hardware-catalog.json),
- change any workflow under `.github/workflows/`, any script under
  `scripts/`, any test under `tests/`, any component under `components/`,
  or any include under `include/`,
- generate firmware, import firmware, or produce a WebFlash artifact,
- mark Sense360 LED as production, preview, compile-only,
  hardware-pending, or blocked in the product catalog,
- add a WebFlash LED build to
  [`config/webflash-builds.json`](../config/webflash-builds.json),
- add `LED` to the Release-One config string
  `Ceiling-POE-VentIQ-RoomIQ` or to the Release-One artifact
  `Sense360-Ceiling-POE-VentIQ-RoomIQ-v1.0.0-stable.bin`,
- change the FanTRIAC HW-005 blocked status,
- change the mains-voltage compliance status for `S360-400` or
  `S360-320`.

It only records:

- what evidence currently exists for an LED-bearing variant,
- which catalog statuses are eligible, given that evidence,
- which artefacts (product YAML, WebFlash wrapper, build-matrix entry,
  release-notes draft, build/release proof) would still need to be
  created before WebFlash could expose Sense360 LED,
- the decision (Option A — keep LED documented only, no catalog entry,
  for **this** PR) and the recommended follow-up PR sequence.

If this document and any source-of-truth document drift, the
source-of-truth document wins and this document must be updated. The
sources of truth this document threads together are listed in
[See also](#see-also).

## Key wording

These four assertions are load-bearing for the decision below. Any
future PR that proposes to weaken any of them must first justify it
against the evidence cited in this document.

- **Verified LED schematic + fixed package pin mapping does not equal
  shippable product.** HW-007 committed `S360-300-R4.pdf` and HW-008
  flipped `S360-300` to `schematic_status: verified` in
  [`config/hardware-catalog.json`](../config/hardware-catalog.json).
  HW-010 reconciled
  [`packages/hardware/led_ring_ceiling.yaml`](../packages/hardware/led_ring_ceiling.yaml)
  to bind `led_data_pin: GPIO38`. Neither of those — individually or
  together — makes Sense360 LED WebFlash-shippable; both are hardware
  / package-level evidence only.
- **LED is still not in current Release-One.** The Release-One
  WebFlash config string `Ceiling-POE-VentIQ-RoomIQ` carries no `LED`
  token, the Release-One product YAML
  [`products/sense360-ceiling-poe-ventiq-roomiq.yaml`](../products/sense360-ceiling-poe-ventiq-roomiq.yaml)
  intentionally omits LED package includes, and the Release-One
  catalog entry in
  [`config/product-catalog.json`](../config/product-catalog.json)
  enumerates `LED` in its `blocked_modules` list.
- **WebFlash must not expose LED until there is a catalog-eligible
  product, artifact, and manifest.** The local taxonomy at
  [`config/webflash-compatibility.json`](../config/webflash-compatibility.json)
  already lists `LED` in `canonical_modules`, but that is a reserved
  taxonomy name (see
  [`docs/webflash-compatibility-taxonomy-audit.md`](webflash-compatibility-taxonomy-audit.md)),
  **not** a shippability claim. WebFlash exposure additionally requires
  a catalog entry with a WebFlash-eligible lifecycle status, a build
  in [`config/webflash-builds.json`](../config/webflash-builds.json), a
  produced and recorded `.bin` artifact, and a WebFlash-side manifest
  (`manifest.json` / `firmware-N.json`) generated by WebFlash from a
  GitHub Release.
- **FanTRIAC remains unrelated and blocked.** This document makes no
  claim about and does not change FanTRIAC. `S360-320` stays
  `cataloged_unverified` in
  [`config/hardware-catalog.json`](../config/hardware-catalog.json);
  the FanTRIAC catalog entry stays `status: blocked`, `blocker:
  HW-005`, `webflash_build_matrix: false`; and the mains-voltage
  compliance tracker [`docs/compliance/mains-voltage-uk-eu-assessment.md`](compliance/mains-voltage-uk-eu-assessment.md)
  remains a separate, additional gate for any future FanTRIAC
  unblock.

## Package mapping status after HW-010

After HW-010 has landed:

- The ceiling LED package
  [`packages/hardware/led_ring_ceiling.yaml`](../packages/hardware/led_ring_ceiling.yaml)
  binds `led_data_pin: GPIO38`, matching the verified `S360-100-R4`
  schematic (`IO38 = LED_DATA → U2A 74LVC1G07 → R8 330 Ω → J3 → S360-300 J1`).
  HW-009 reclassified the `LED_DATA / Sense360 LED` mapping row from
  `needs-package-change` to `confirmed-ok` after HW-010 landed (see
  [`docs/hardware/firmware-package-mapping-audit.md`](hardware/firmware-package-mapping-audit.md)).
- The wall LED package
  [`packages/hardware/led_ring_wall.yaml`](../packages/hardware/led_ring_wall.yaml)
  is unchanged. It still uses `led_data_pin: GPIO48`. No Core-side
  schematic is committed proving this is the correct `LED_DATA` path
  on the wall variant; HW-010 deliberately did not touch this package.
  Its only catalog consumers (`sense360-core-wall.yaml` and friends)
  are `legacy-compatible`, off the WebFlash track.
- The legacy S3 Core package
  [`packages/hardware/sense360_core_ceiling_s3.yaml`](../packages/hardware/sense360_core_ceiling_s3.yaml)
  is unchanged. It still uses `led_data_pin: GPIO14`. No Core-side
  schematic is committed for the S3 variant; there is no product YAML
  in this repo that consumes this package. HW-010 deliberately did not
  touch it.
- HW-010 added
  [`tests/test_led_package_mapping.py`](../tests/test_led_package_mapping.py)
  which locks in: ceiling LED package = `GPIO38`; Release-One YAML
  does **not** `!include` `led_ring_ceiling.yaml`; no `LED` token in
  any build in
  [`config/webflash-builds.json`](../config/webflash-builds.json);
  Release-One catalog entry still lists `LED` and `FanTRIAC` in
  `blocked_modules`; FanTRIAC entry stays `status: blocked`,
  `blocker: HW-005`, `webflash_build_matrix: false`.

**HW-010 changed nothing about WebFlash shippability for Sense360 LED.**
It is a single-package edit that aligns one of three LED packages with
schematic evidence. A future LED-bearing config requires its own
product YAML, its own WebFlash wrapper, its own catalog entry, its own
build-matrix entry, its own release-notes draft, and its own
build/release proof — none of which HW-010 produced.

## Config-string options

There is one canonical config-string form for a LED-bearing
ceiling-PoE-VentIQ-RoomIQ variant. It follows the existing module
ordering rule in
[`docs/webflash-contract.md`](webflash-contract.md) §7 and is also the
example string used in
[`docs/product-onboarding.md`](product-onboarding.md),
[`docs/webflash-compatibility-taxonomy-audit.md`](webflash-compatibility-taxonomy-audit.md),
[`docs/release-one-hardware-audit.md` Detail / Sense360 LED policy](release-one-hardware-audit.md),
[`docs/hardware/s360-300-r4-led.md`](hardware/s360-300-r4-led.md),
[`docs/hardware/firmware-package-mapping-audit.md`](hardware/firmware-package-mapping-audit.md),
[`docs/cleanup-audit.md`](cleanup-audit.md), and
[`tests/test_webflash_compatibility.py`](../tests/test_webflash_compatibility.py).

```text
Ceiling-POE-VentIQ-RoomIQ-LED
```

Grammar check against
[`config/webflash-compatibility.json`](../config/webflash-compatibility.json):

| Token | Slot | Allowed? | Notes |
|---|---|---|---|
| `Ceiling` | `canonical_mounting` | Yes | Only canonical mounting. |
| `POE` | `canonical_power` | Yes | Same power slot as Release-One. |
| `VentIQ` | `canonical_modules` air-quality slot | Yes | Mutex with `AirIQ`; not triggered here. |
| `RoomIQ` | `canonical_modules` room-sense slot | Yes | Pairs with `AirIQ` or `VentIQ` per existing rules. |
| `LED` | `canonical_modules` indicator slot | Yes | Already a canonical module; no mutex, no forbidden-token collision, no fan-driver rule triggered. |

No alternative ordering is proposed. `Ceiling-POE-VentIQ-LED-RoomIQ`,
`Ceiling-POE-VentIQ-RoomIQ-LED-…`, or any other permutation is **not**
considered by this document — the existing audit docs and tests
already pin the `Ceiling-POE-VentIQ-RoomIQ-LED` form.

Adding `LED` to the **Release-One** config string is explicitly **not
considered**. Release-One's config string and artifact are fixed by
[`docs/release-one.md`](release-one.md) and verified by
[`tests/test_webflash_artifact_naming.py`](../tests/test_webflash_artifact_naming.py).
An LED-bearing variant is a sibling product, not a Release-One edit.

## Catalog status options

The catalog at
[`config/product-catalog.json`](../config/product-catalog.json)
declares the following `lifecycle_statuses`:

| Status | Applicable to an LED-bearing variant today? | Why / why not |
|---|---|---|
| `production` | **No.** | Requires `config_string`, `product_yaml`, `webflash_wrapper`, `artifact_name`, `version`, `channel`, `hardware_status`, `hardware`, `modules`, `webflash_build_matrix: true`, an existing wrapper file, a matching build in [`config/webflash-builds.json`](../config/webflash-builds.json), and a recorded build/release proof per [`docs/product-onboarding.md` Required evidence before production](product-onboarding.md#required-evidence-before-production). None of those artefacts exists for an LED-bearing variant. Additionally, the only `production` slot today is Release-One's `Ceiling-POE-VentIQ-RoomIQ`. |
| `preview` | **No** (yet). | Requires `config_string`, `product_yaml`, `hardware_status`; if `webflash_build_matrix: true` it additionally requires an on-disk `webflash_wrapper`; `channel` must not be `stable`; `config_string` must not be in `release_one_required_configs`. No LED-bearing product YAML exists, so `preview` cannot be set today even with `webflash_build_matrix: false`. |
| `compile-only` | **No** (yet). | Catalog rule block requires the entry to be self-consistent against a YAML on disk and pass the consistency validator. Adding a `compile-only` entry without first creating the product YAML would fail [`scripts/validate_product_catalog_consistency.py`](../scripts/validate_product_catalog_consistency.py). Therefore a `compile-only` entry must follow a YAML-creation step, not precede it. |
| `hardware-pending` | **No.** | The hardware evidence is *present* (HW-007 / HW-008 / HW-009 / HW-010) — `S360-300` is `documented` and `schematic_status: verified`. The blocker for an LED-bearing variant is **not** hardware; it is **product packaging** (no YAML, no wrapper, no catalog entry, no build, no release-notes draft, no build/release proof). `hardware-pending` is the wrong status. |
| `blocked` | **No.** | Requires a named `blocker` (e.g. `HW-005` for FanTRIAC) and a `reason`. No comparable blocker exists for Sense360 LED today: HW-007 / HW-008 / HW-010 have all landed and the schematic evidence is verified. Using `blocked` would be misleading. |
| `legacy-compatible` | **No.** | Used for pre-WebFlash product YAMLs retained for manual / custom users. Not applicable to a new WebFlash-track variant. |
| `deprecated` / `removed` | **No.** | Lifecycle states for products that previously shipped. There is no LED-bearing variant in any prior state. |
| *(no catalog entry — deferred)* | **Yes — pick now.** | No row is added to [`config/product-catalog.json`](../config/product-catalog.json) by this PR. The decision is recorded in this document only. A future scoped PR may add a `compile-only` entry after the LED-bearing product YAML lands. |

## Product YAML / wrapper requirements

Before any LED-bearing catalog entry can land, the following artefacts
must exist on disk. Each is its own scoped follow-up PR (see
[Follow-up PR sequence](#follow-up-pr-sequence)). None of them is
created by this PR.

1. **Product YAML** —
   `products/sense360-ceiling-poe-ventiq-roomiq-led.yaml`. Composed
   from the same Core ceiling + PoE PSU + VentIQ + RoomIQ package
   stack as the Release-One YAML, **plus** the ceiling LED package
   [`packages/hardware/led_ring_ceiling.yaml`](../packages/hardware/led_ring_ceiling.yaml).
   The Release-One YAML
   [`products/sense360-ceiling-poe-ventiq-roomiq.yaml`](../products/sense360-ceiling-poe-ventiq-roomiq.yaml)
   must not be edited — adding the LED package to Release-One would
   change the binary built from the production config string and
   violate
   [`tests/test_led_package_mapping.py::test_release_one_does_not_include_led_ceiling_package`](../tests/test_led_package_mapping.py).
2. **WebFlash wrapper** —
   `products/webflash/ceiling-poe-ventiq-roomiq-led.yaml`. Required
   only if the catalog entry sets `webflash_build_matrix: true`. The
   wrapper basename (without `.yaml`) must equal the `config_string`
   lower-cased, per the validator rule in
   [`scripts/validate_product_catalog_consistency.py`](../scripts/validate_product_catalog_consistency.py).
3. **Catalog entry** in
   [`config/product-catalog.json`](../config/product-catalog.json),
   with `config_string: "Ceiling-POE-VentIQ-RoomIQ-LED"` and a
   lifecycle status appropriate to which prior gates have been
   cleared. The expected order is `compile-only` → `preview` (on a
   non-`stable` channel) → `production` (only after a recorded build /
   release proof).
4. **Build-matrix entry** in
   [`config/webflash-builds.json`](../config/webflash-builds.json),
   only after the catalog entry is `preview` or `production`. The
   channel must be non-`stable` for `preview`. Validated by
   [`tests/validate_webflash_builds.py`](../tests/validate_webflash_builds.py).
5. **Release-notes draft** generated by
   [`scripts/generate_webflash_release_notes.py`](../scripts/generate_webflash_release_notes.py).
   The generator currently emits `Sense360 LED` as a Known-Issues
   exclusion for every build; it would need a scoped change so that
   for an LED-bearing build, Sense360 LED moves into Features for
   that build only (and remains in Known-Issues for the Release-One
   build, which stays LED-less). That change is **not** in this PR.
6. **Test update.**
   [`tests/test_led_package_mapping.py::WebflashBuildsLedExclusionTests::test_no_led_token_in_any_build`](../tests/test_led_package_mapping.py)
   today asserts that no build has `LED` in its config string or
   artifact name. When the LED-bearing build is added, that test
   must be re-scoped to the **Release-One** build only (i.e. the
   Release-One build must remain LED-less, but a sibling LED-bearing
   build is allowed to carry the `LED` token). The class
   `LedCeilingPackageMappingTests::test_release_one_does_not_include_led_ceiling_package`
   and `FanTRIACStatusUnaffectedTests` invariants stay as-is. That
   test edit is **not** in this PR.
7. **Build / release proof.** Required only for promotion to
   `preview` (recorded proof of a successful CI build on a non-`stable`
   channel) or `production` (full proof equivalent to the Release-One
   row in
   [`docs/release-one.md` Proof of build (recorded)](release-one.md#proof-of-build-recorded)).

## Recommendation

**Option A for this PR. Option C later, not now.**

That is:

- **For this PR:** no catalog entry, no product YAML, no WebFlash
  wrapper, no build-matrix entry, no release-notes draft. This
  document records the decision, the option matrix, the follow-up PR
  sequence, and the do-not-change list.
- **Eventually:** Sense360 LED may be considered for `preview` only
  after a dedicated LED-bearing product YAML and WebFlash wrapper are
  created and compile-tested. The natural first catalog state for an

  equivalent to Release-One's).
- **Never (without an explicit upstream decision):** adding `LED` to
  the Release-One stable config string. An LED-bearing variant is a
  sibling product, not a Release-One edit.
